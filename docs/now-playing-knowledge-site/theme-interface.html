theme interface

Purpose

This page documents now-playing/theme.html, the shell-theme editor in the now-playing ecosystem.

This page is intentionally implementation-aware, because theme.html appears to be a live, client-side theme editor rather than a conventional server-backed settings page. It is designed for editing core shell color tokens, previewing changes live, saving/loading presets, importing/exporting theme JSON, and synchronizing with the app shell through postMessage.

Why this page matters

theme.html is important because it reveals that the shell’s visual configuration is not just hard-coded styling. It is an editable token system with persistent local state, named presets, live preview behavior, and shell/editor synchronization messages.

Important file

Primary file:

  • now-playing/theme.html

Related pages include configuration-and-diagnostics-interfaces.md, desktop-browser-interface.md, display-interface.md, and config-interface.md.

High-level role

A good current interpretation is that theme.html is the operator/designer-facing shell-theme editor, edits tokenized visual state for the shell and related UI surfaces, and is primarily client-side and stateful in localStorage rather than runtime-API-centered.

Major UI areas visible in the page

1. Token grid / color picker cards

The main editor surface includes #tokenGrid, one card per theme token, color swatches and hidden native color pickers, and text inputs for raw token values.

2. Preset controls

Observed preset controls include #presetSelect, #savePresetBtn, and #deletePresetBtn.

3. Import/export controls

Observed controls include #importBtn, #exportBtn, and an export modal with #exportJsonText, #exportCopyBtn, and #exportCloseBtn.

4. Save/reset controls

Observed controls include #saveBtn, #resetBtn, and #statusMsg.

Core storage model

One of the most important architectural facts in theme.html is that it is localStorage-centered. Observed keys include nowplaying.themeTokens.v1, nowplaying.themePresets.v1, and nowplaying.themeActivePreset.v1.

Theme token model

The page defines a large token set with defaults across shell/background, text, rail/frame, tab, hero/card, and pill/progress categories. Observed examples include --theme-bg, --theme-text, --theme-text-secondary, --theme-rail-bg, --theme-frame-fill, --theme-rail-border, --theme-tab-bg, --theme-tab-active-bg, --theme-hero-card-bg, --theme-hero-card-border, --theme-picker-card-bg, --theme-pill-border, and --theme-pill-glow.

Key functions / logic blocks

load()

Loads current theme token state from nowplaying.themeTokens.v1 and merges it with defaults.

loadPresets() / savePresets()

Manage named preset storage via nowplaying.themePresets.v1, including starter-preset backfill and user-preserving merge behavior.

render()

Builds the token editor UI, wiring cards to color pickers and text inputs and pushing changes live as the user edits.

push()

Applies token changes to the current document and broadcasts type: 'np-theme-updated' with current token state.

applyThemeEditorZones() and applyPickerTileBg()

Keep the editor itself visually synced with the current token state.

Preset-selection behavior

The page includes a meaningful preset workflow: selecting a preset from #presetSelect, applying its tokens to local state, saving a preset under a prompted name, deleting a selected preset, and tracking an active preset name.

Import/export behavior

The page includes an explicit JSON portability model: export current theme to JSON, show it in a modal text area, copy it to the clipboard, import theme JSON via prompt, and normalize payload format through normalizeThemePayload(...) and buildThemePayload(...).

Live shell synchronization

Observed message types include np-theme-updated, np-theme-sync, and np-theme-load-preset, strongly suggesting the editor can push theme state outward to a shell or parent and the shell can push mode/preset state back into the editor.

Reset behavior

Current visible behavior suggests reset returns the editor to Slate Medium, implying the starter preset set includes a canonical default baseline used as the reset target.

Starter presets

The page defines a fairly large starter preset set, including Slate Medium, Chill Blue, Monotone Gray, Warm Parchment, Muted Merlot, Abyss Graphite, Matrix, Blue Neon, and Red Neon.

Architectural interpretation

A good current interpretation is that theme.html is a client-side design-token editor for the shell, persists local theme state and presets in the browser, propagates changes live through DOM application and postMessage, and is aimed more at shell/presentation customization than at runtime-server configuration.

Current status

At the moment, this page already has enough evidence to be treated as the main shell-theme editing surface in the project. It is not just a color picker; it is a local preset system, token editor, export/import tool, and live shell synchronization surface.
