playback authority by mode

Purpose

This page documents how playback truth changes depending on the active playback mode.

It exists because there is not one universal source of truth for current song, current art, current status, and what should be shown to the user. The authoritative path depends on the active mode.

Core model

A good current interpretation is:

  • playback authority is mode-dependent
  • MPD remains important, but it is not the only truth source that matters for visible behavior
  • art and current-song interpretation can differ by mode even when transport still exists underneath
  • app-host routes often translate multiple underlying truths into one visible result

This page is the companion to queue-and-playback-model.md.

Mode 1: local file playback

A good current interpretation is:

  • local file playback is the clearest case where MPD truth and local library/file truth align most directly
  • current-song and playback state are strongly grounded in MPD/local-file playback reality
  • art authority often resolves through moOde/local-file cover-art paths such as moOde coverart.php and local/current-art cache-backed fallbacks where relevant

Mode 2: AirPlay

A good current interpretation is:

  • AirPlay playback does not behave like plain local-file playback
  • AirPlay metadata and art are strongly influenced by the AirPlay metadata path
  • current visible art/status often depends on aplmeta.txt, AirPlay-specific cover/metadata readers, and moOde current-song/status paths as interpreted for AirPlay

Important local-environment note: Brian’s setup already has local hardening/override behavior around airplay-json, watchdogs, and metadata readers.

Mode 3: UPnP

A good current interpretation is:

  • UPnP is its own branch, not just a generic stream case
  • visible truth may combine moOde current-song/status, UPnP-specific resolution logic, and attempts to resolve apparently stream-like items back toward local-file cover/art truth when possible

Mode 4: radio / stream

A good current interpretation is:

  • radio/stream is one of the most nuanced authority paths in the project
  • raw stream metadata is often not trusted directly as final truth
  • visible output may be shaped by station holdback policy, metadata cleanup/normalization, enrichment lookup attempts, guardrails/suppression, station-logo fallback, and cache-backed/current-art fallback paths

Important principle: radio/stream behavior is not “whatever raw metadata says.” It is an interpreted mode with its own quality-control logic.

Mode 5: unresolved / fallback-driven case

Sometimes the system cannot confidently resolve a better authority path. In those cases:

  • fallback behavior becomes authoritative enough for presentation
  • cache-backed paths such as /art/current.jpg may become the practical visible truth
  • the system may preserve a last-known-good result rather than aggressively clearing the UI

Cross-mode principles

  • Do not assume one universal authority.
  • Visible truth is often app-mediated.
  • Art authority is especially mode-sensitive.
  • Troubleshooting should start with mode classification.

Suggested troubleshooting order by mode

  • Local file: check MPD current song/queue, local file/library identity, moOde/local-file art path.
  • AirPlay: check AirPlay metadata path health, aplmeta.txt, AirPlay cover/metadata pipeline, local overrides/watchdog behavior.
  • UPnP: check moOde current-song/status, UPnP-specific resolution logic, and whether fallback to local-file truth is expected.
  • Radio/stream: check raw metadata quality, holdback logic, cleanup/normalization, enrichment guardrails, and station-logo/fallback art path.
  • Unresolved/fallback: check last-good-render preservation and cache-backed art behavior.

Current status

This page gives the wiki one of the most important missing ideas: playback truth, art truth, and visible output are mode-dependent, and debugging should begin with mode classification rather than random file searching.
