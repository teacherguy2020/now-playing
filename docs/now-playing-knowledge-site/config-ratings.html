config ratings

Purpose

This page documents the ratings-related portion of now-playing/config.html.

It exists because ratings in this project are not just a small preference toggle. The Config page exposes a real ratings-management subsystem built around the MPD sticker database.

Main visible controls

  • featureRatings
  • checkRatingsDbBtn
  • backupRatingsDbBtn
  • ratingsBackupSelect
  • restoreRatingsDbBtn
  • ratingsDbStatus

Ratings feature enablement

Config loads and saves ratings through features.ratings, but the visible card makes clear that enablement is only one part of the story.

Ratings DB status check

checkRatingsDb() sends:

  • GET /config/ratings/sticker-status

with x-track-key, then reports whether the sticker DB exists, where it lives, and how large it is.

Important nuance: the UI explicitly treats “not found yet” as a normal possible state before the first rating write.

Ratings backup listing

loadRatingsBackups() sends:

  • GET /config/ratings/sticker-backups

with x-track-key and populates the backup selector with path, name, and size.

Ratings backup creation

backupRatingsDb() sends:

  • POST /config/ratings/sticker-backup

with x-track-key, reports the backup path, and refreshes the backup list.

Ratings restore workflow

restoreRatingsDb():

  • requires a selected backup
  • asks for explicit confirmation
  • warns that the current sticker DB will be overwritten
  • sends POST /config/ratings/sticker-restore with { backupPath }

That makes restore an intentionally guarded maintenance action.

Important endpoints

  • GET /config/ratings/sticker-status
  • GET /config/ratings/sticker-backups
  • POST /config/ratings/sticker-backup
  • POST /config/ratings/sticker-restore

Track-key protection

All ratings maintenance actions are track-key protected using x-track-key. That means the ratings DB workflows are treated as privileged maintenance actions.

User/operator workflow model

  1. Enable Ratings
  2. Check Ratings DB
  3. Create a backup before risky work
  4. Select and restore a backup only when needed
  5. Verify behavior afterward

Architectural interpretation

Ratings should be documented as both:

  • a feature flag
  • a persistence and backup/restore subsystem

That is the important takeaway.
