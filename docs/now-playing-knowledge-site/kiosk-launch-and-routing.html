kiosk launch and routing

Purpose

This page documents how kiosk mode is actually entered and handed off in the current repo.

Direct file inspection shows that much of the kiosk path is profile resolution, localStorage synchronization, redirect handoff into controller.html, and very thin kiosk-named alias pages that immediately redirect into controller-family pages.

kiosk.html — profile-sync and redirect bridge

kiosk.html has no visible kiosk UI of its own. It is essentially a script-only handoff page.

It explicitly uses these localStorage keys:

  • nowplaying.kiosk.profile.v1
  • nowplaying.mobile.profile.v2

It reads incoming query params including:

  • theme
  • colorPreset
  • recentSource
  • primaryThemeColor
  • secondaryThemeColor
  • primaryTextColor
  • secondaryTextColor

Then it resolves effective kiosk settings, writes a normalized kiosk profile, synchronizes a controller/mobile profile payload, and redirects to:

  • controller.html?kiosk=1&preview=1&rev=20260329-motion-art-fix...

The redirected URL also forwards theme/color/recent-source/custom-color parameters when present.

Important: the inline comment shows rev=20260329-motion-art-fix is there to cache-bust the controller shell so kiosk navigation fixes apply immediately.

Best interpretation: kiosk.html is a profile-sync + redirect bridge, not the live kiosk shell.

controller.html — actual kiosk-backed destination

Direct inspection confirms that controller.html contains real kiosk-specific behavior, including:

  • preview=1 handling
  • kiosk detection from kiosk=1, profile=1280x400, or layout=1280x400
  • body.kiosk1280 and body.kioskEditor modes
  • #kioskPane and #kioskPaneFrame
  • message handling for np-kiosk-hide-pane
  • query-driven theme/profile application

So the redirected destination really does provide the controller-backed kiosk shell and pane infrastructure.

Kiosk alias pages — pure redirect shims into controller surfaces

Direct file inspection shows that multiple kiosk-named pages are literal one-line redirect aliases preserving location.search:

  • kiosk-albums.html → controller-albums.html
  • kiosk-artists.html → controller-artists.html
  • kiosk-playlists.html → controller-playlists.html
  • kiosk-radio.html → controller-radio.html
  • kiosk-podcasts.html → controller-podcasts.html
  • kiosk-genres.html → controller-genres.html
  • kiosk-queue.html → controller-queue.html
  • kiosk-queue-wizard.html → controller-queue-wizard.html
  • kiosk-now-playing.html → controller-now-playing.html

These pages are aliases, not independent kiosk UI implementations.

controller-kiosk.html — separate scaffold/prototype path

controller-kiosk.html is clearly separate from the main redirect bridge model. It is a standalone scaffold page with:

  • fixed 1280×400 layout
  • three-pane structure
  • hardcoded source buttons
  • a Now Playing panel fed from /now-playing
  • placeholder append/crop/replace actions
  • phase-1 / phase-2 scaffold language in the file itself

Best interpretation: this is a scaffold/prototype path, not the main live kiosk handoff path.

Working routing model

  1. Open kiosk.html
  2. Resolve incoming theme/recent-source/custom-color parameters
  3. Merge with saved kiosk profile
  4. Write normalized kiosk profile
  5. Sync controller/mobile profile state
  6. Redirect into controller.html?kiosk=1&preview=1&rev=...
  7. Let controller.html provide the actual kiosk-backed shell behavior
  8. Use kiosk-named alias pages to route child surfaces into controller-family views

Key distinctions to preserve

  • kiosk.html vs controller.html: redirect bridge vs actual kiosk-backed destination
  • kiosk alias pages vs real implementations: many are aliases only
  • controller-kiosk.html vs main kiosk path: scaffold/prototype vs live handoff path

Current status

This page is now much less guessy: the main kiosk entrypoint is a profile-sync + redirect bridge, the live kiosk shell is controller-backed, many kiosk-named files are aliases, and the separate scaffold path is controller-kiosk.html.
