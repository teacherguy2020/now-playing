library health interface

Purpose

This page documents now-playing/library-health.html, the operator-facing library audit and maintenance console in the now-playing ecosystem.

This page is intentionally implementation-aware, because library-health.html is clearly not just a passive report. It appears to be a maintenance workbench for scanning library health, inspecting animated art cache state, inspecting album metadata, suggesting and applying performer metadata, overwriting tags across albums/tracks, cleaning up collaborative/featured artist tags, and triggering a moOde library update.

Why this page matters

library-health.html appears to be one of the most important maintenance pages in the whole operator branch. It bridges media-library structure, cached/derived visual assets, album metadata quality, cleanup workflows, and moOde library state.

Important files

  • Primary page: now-playing/library-health.html
  • Important supporting script: now-playing/scripts/library-health.js

Related pages include configuration-and-diagnostics-interfaces.md, config-interface.md, and tablet-interface.md.

High-level role

A good current interpretation is that library-health.html is the operator’s media-library audit and repair surface, brings together multiple maintenance workflows that would otherwise be scattered, and is one of the strongest bridges between library data, metadata quality, and visible display outcomes.

Shell / entry behavior

Like other operator pages, library-health.html contains shell-redirect behavior. It also contains additional embedded-shell stripping logic that removes shell chrome when hosted inside another page context.

Major UI areas visible in the page

1. Library scan / summary area

Top-level operator actions include #refreshScan, #updateMoodeLibraryBtn, a cards summary region via #cards, and status regions for scan and moOde update feedback.

2. Animated art cache section

The page includes #animatedArtCacheCard, #animatedArtSummary, #animatedArtCacheList, #animatedArtPreview, and motion/no-motion/all filter radios.

3. Album metadata inspector

This is one of the richest sub-features in the page, with album pick/inspect controls, performer suggestion/apply controls, overview/art/genre/cleanup tabs, performer input area, tag-overwrite tools, and metadata output/status regions.

4. Tag overwrite workflow

Visible elements include #albumTagKey, #albumTagValue, #albumTagClearEmpty, #albumTagFillAllBtn, and #albumTagApplyBtn.

5. Featured-artist cleanup workflow

The page includes a dedicated cleanup area for collaborative feat-style artist tag cleanup, with controls like #featAlbumPick, #featScanBtn, #featApplyBtn, #featNormalizeAll, #featStatus, and #featOut.

Supporting script as the likely logic center

The page includes scripts/library-health.js, which strongly suggests most of the scanning, album inspection, animated-art listing, metadata mutation, and cleanup logic lives there rather than inline.

Use it when the task is not just about Library Health as a surface, but about a specific region such as the scan summary, animated-art cache section, album metadata inspector, performer workflow, tag overwrite tool, feat cleanup workflow, or moOde library update action.

Relationship to the media-library and artwork branches

This page sits at the intersection of media library representation, album metadata quality, artwork and animated-art caches, cleanup/curation workflows, and moOde library synchronization.

moOde library update role

The visible presence of #updateMoodeLibraryBtn makes an important point: this page is not only inspecting local derived state, it is also involved in driving or triggering moOde library refresh/update behavior.

Album-inspection workflow model

A useful current model is: choose album context, inspect metadata, review art/genre/cleanup subviews, suggest performers or edit them manually, apply performer changes, optionally use tag-overwrite tools, then optionally run feat-cleanup tools.

Architectural interpretation

A good current interpretation is that library-health.html is the operator-side maintenance console for library integrity and metadata quality, tying together scanning, inspection, cleanup, and sync-oriented actions. It likely plays a significant role in keeping the visible system sane across controller, display, and artwork surfaces.

Things still to verify

Future deeper verification should clarify what exact endpoints scripts/library-health.js calls for scan, inspect, suggest, apply, and cleanup actions; how animated art cache data is sourced and refreshed; how album tag overwrite is applied; whether performer suggestions are heuristic, metadata-backed, or externally enriched; what exactly the moOde library update action triggers on the backend; and which workflows are read-only vs state-mutating vs restart-sensitive.

library-health-page-anatomy.md matters when the real question is not only “what is library-health.html?” but “which region inside library-health.html actually owns the thing I need to change?”

Current status

At the moment, this page already has enough evidence to be treated as a core maintenance surface. It is not merely a status page; it is a practical library audit, metadata repair, artwork/cache inspection, and moOde-sync workbench.
