repo coverage notes

Purpose

This page records the current coverage state of the now-playing wiki against the actual now-playing/ repository.

It is a coverage map, not a promise that every file has its own wiki entry.

Current overall assessment

A good current assessment is:

  • the wiki now strongly covers the surface/interface layer and the main architectural/engineering branches
  • it does not try to be a file-by-file encyclopedia of the entire repo
  • that is mostly the right tradeoff for the project’s current maturity

Practical summary:

  • all important HTML surface families are covered
  • major architectural branches are covered
  • route/service/directory coverage is strategic rather than exhaustive
  • low-value packaging/vendor/artifact areas are intentionally not mapped deeply

Strongly covered areas

Top-level HTML surface layer

This is one of the strongest areas in the wiki. Important page families across controller/browser/display/kiosk/operator surfaces are now broadly covered.

config/

Coverage level: strong. The config branch is one of the strongest branches in the wiki.

API / playback / queue / ownership branch

Coverage level: strong. This was the main Phase 2 target and is now substantially complete.

Runtime / ops / local-environment branch

Coverage level: strong. Runtime reality, host boundaries, restart sensitivity, and verification paths are now explicitly documented.

Strategically covered areas

src/

Coverage level: strategically covered. The wiki is useful for navigating src/, but it is not yet a full file-level map.

integrations/

Coverage level: strategically covered. Integration boundaries are well represented conceptually and operationally.

ops/

Coverage level: strategically covered. Operational importance and override relevance are documented without deeply enumerating every file.

alexa/

Coverage level: strategically covered. Alexa as a branch is represented meaningfully, but not every file is described individually.

scripts/

Coverage level: strategically covered. Important script ownership appears where relevant, but there is no dedicated scripts map page yet.

Lightly covered areas

  • styles/
  • skins/
  • assets/
  • lambda_bundle/
  • lambda_upload/
  • notes/
  • exports/
  • icons/

These are represented only lightly or indirectly.

Intentionally low-value or uncovered areas

  • node_modules/ — intentionally uncovered
  • packaging artifacts, archives, and incidental assets — intentionally low-value
  • miscellaneous top-level support files — intentionally low-value unless one becomes operationally important

Recently improved gap areas

Recent additions strengthened coverage for source-doc areas that were previously underrepresented:

  • install and validation workflow now has a dedicated support page
  • style/token naming and background invariants now have a dedicated support page
  • display enhancement now has a concrete Peppy/Player operational page
  • tablet recents and Last.fm row-source behavior now have a dedicated child page
  • AirPlay metadata hardening now has a dedicated support page

These additions improve preservation of operational contracts and fragile design-system rules without trying to turn the wiki into a file-by-file encyclopedia.

Highest-value future repo-coverage expansions

  • deeper src/ mapping, especially src/routes/, src/services/, and src/lib/
  • scripts/ mapping for diagnostics/display/library-health/theme/runtime behavior
  • styles/ + skins/ display asset branch if display/render work becomes active again

Current status

This page gives the wiki a self-aware coverage map, making the knowledge base more honest and more useful.
