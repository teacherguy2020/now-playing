display issue triage runbook

Purpose

This runbook gives a practical triage order for display- and presentation-side problems in now-playing.

It is meant for issues like wrong display surfaces, Player/Peppy/Visualizer/Kiosk confusion, iframe-vs-top-level differences, wrong routed page vs wrong final render target, moOde browser target mismatches, and visible presentation problems that may actually be route/runtime/integration problems.

Core rule

When a display issue appears, triage in this order:

  1. identify the actual surface/mode
  2. identify the primary page owner
  3. decide whether the issue is page-owned, route-owned, or host/runtime-owned
  4. check historically fragile display paths
  5. verify the right host-side effect

Step 1: identify the actual display surface or mode

  • Browser/TV display surface — usually display.html or index.html
  • Player display mode — player-render.html as final render surface, player.html as designer/preview surface
  • Peppy display mode — usually peppy.html, possibly a Peppy-skin-specific issue
  • Visualizer display mode — usually visualizer.html or wrapper hosts
  • Kiosk presentation/shell mode — kiosk.html, controller-backed kiosk flows, or kiosk-designer.html for authoring/push issues

Step 2: identify the primary page owner

Start with display.html/index.html for classic display routing, player-render.html for Player mode, peppy.html for Peppy mode, and kiosk.html or controller-backed kiosk surfaces for Kiosk.

Important guardrail: always ask whether the page is the router, wrapper, designer, or actual final render target.

Step 3: decide what layer most likely owns the symptom

  • Page-owned — wrong layout/visible behavior on one page or context
  • Route-backed — page renders but art/text/mode/state is wrong
  • Host/runtime-owned — route says success but moOde box still shows the wrong thing

Step 4: check the historically fragile display paths

  • iframe vs top-level rendering — start with index.html, display.html, scripts/index-ui.js, styles/index1080.css
  • routed page vs final render target confusion — especially important for display.html, player.html, player-render.html, and wrapper pages
  • app-host vs moOde-host mismatch — start with runtime-admin routes, local environment, and real moOde host verification
  • mode-sensitive art/display truth — start with playback mode classification and playback-authority-by-mode

Step 5: decide which host to verify

  • App host (10.0.0.4) — page logic, route responses, routed mode selection, support data
  • moOde host (10.0.0.254) — display box still wrong, runtime-admin actions seem ineffective, wake/display/browser-target behavior

Step 6: use the minimum verification set that matches the symptom

  • Wrong mode/page showing: verify router/launcher choice first, final render target second, real destination surface last
  • Art/text wrong but page structure fine: verify playback mode, support route output, display surface refresh, fallback/last-good expectations
  • Preview/designer differs from live target: verify designer/helper surface, final render target, and real pushed live target
  • Runtime-admin display push ineffective: verify route response, status endpoint, real moOde-host visible result, and local override possibility

Step 7: declare success only when the visible destination is correct

A display issue is solved only when the correct routed/rendered destination surface is identified, the right layer has been checked, and the real visible destination now behaves correctly.
