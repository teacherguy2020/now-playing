config alexa setup

Purpose

This page documents the Alexa setup portion of now-playing/config.html.

It exists because the Config page owns the provisioning/setup side of the Alexa integration, while the separate alexa.html page owns the correction/review side.

Main visible controls

  • featureAlexaSkill
  • alexaDomain
  • checkAlexaDomainBtn
  • alexaDomainCheckStatus
  • alexaRouteWebhookUrl
  • alexaDomainLight

Wider shell-level feedback also uses alexaHint and alexaPill.

Alexa enablement

Config loads and saves Alexa enablement through alexa.enabled.

Public domain configuration

The public domain is stored as alexa.publicDomain. The page uses its presence/absence together with enablement state to drive the Alexa hint/pill/light state.

Route webhook URL

alexaRouteWebhookUrl is saved as alexa.routeWebhookUrl and is explicitly described in the UI as being used by “Route to Alexa” buttons in Live Queue / Hero Transport.

Domain reachability check

checkAlexaDomain() sends:

  • POST /config/alexa/check-domain

with x-track-key and { domain }, then reports whether the domain is reachable and updates the Alexa pill/light state accordingly.

Automatic verification behavior

When Alexa is enabled, a domain exists, and a track key is available, the page auto-runs the domain check on load. So verification is proactive, not only manual.

Important boundary

The Config page explicitly says:

  • Config is the source of truth for enable/domain setup
  • alexa-interface.md / alexa.html focuses on corrections and misheard review

That split is one of the main things this page preserves.

Current status

This block is best understood as the Alexa provisioning/control-plane layer for:

  • enablement
  • public domain
  • route webhook setup
  • reachability verification
  • setup-state feedback
