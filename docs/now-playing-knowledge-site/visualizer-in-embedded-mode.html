visualizer in embedded mode

Purpose

This page documents what visualizer.html actually is, and then explains how it behaves when embedded inside controller/kiosk flows.

A better current interpretation is:

  • visualizer.html is a reactive visual engine
  • it exposes a collection of scene/model presets
  • those visuals react to live music-driven feature data such as loudness/volume envelopes, band/spectrum values, flux/pulse-style event signals, and related motion drivers
  • embedded mode is one operating context for that engine, not the definition of what it is

What visualizer.html actually is

Direct file inspection shows that visualizer.html is a multi-scene reactive visual surface with:

  • a canvas-based animation loop
  • preset/model selection
  • energy/motion/glow controls
  • local saved presets
  • moOde push behavior
  • different operating modes for designer, embedded, and kiosk-fullscreen use

It is fundamentally a music-reactive visual system, not just a page that happens to be embedded.

Core reactive model

Direct file inspection shows vocabulary and state around live motion/reactivity such as bandVals, loudness/volume envelope state, event-style triggers like spectralFlux and pulse, requestAnimationFrame, and canvas-driven rendering.

The core job of the page is to take continuously changing music-driven feature state, feed it into a chosen visual scene/model, and render a reactive animation frame-by-frame.

Scene / model family

Direct file inspection shows multiple visual scene/model names and related preset vocabulary. Observed examples include aeon, mobius, orb, cube, bloom field, fireworks, sphere, moire, and ambient.

The important current truth is that the visualizer is not one effect. It is a family of visual models/scenes with different rendering behavior.

Preset / control layer

Direct file inspection shows a substantial preset/configuration layer with controls such as presetSel, savedPresetSel, energySel, motionSel, glowSel, applyBtn, savePresetBtn, deletePresetBtn, exportPresetBtn, importPresetBtn, and pushMoodeBtn.

Saved presets are stored under nowplaying.visualizer.presets.v1.

Query-parameter inputs

visualizer.html reads and reacts to query parameters including embedded, pure, kiosk, ui, preset, energy, motion, and glow.

What embedded mode changes

Embedded mode sets embeddedOn. Current file behavior shows that embedded mode forces pure mode, and pure mode hides HUD/controls. So embedded mode changes how the reactive visual engine is hosted, not what the visualizer fundamentally is.

Embedded visualizer → fullscreen kiosk handoff

In embedded mode, a pointer/tap handler removes embedded and pure, sets kiosk=1 and ui=0, and navigates the top window to the fullscreen kiosk visualizer target.

Fullscreen kiosk visualizer → controller return path

When kioskOn && !embeddedOn && !uiOn, a pointer/tap handler returns to:

  • controller.html?kiosk=1&preview=1&open=visualizer.html&vpreset=...&venergy=...&vmotion=...&vglow=...

This creates a real round-trip between embedded pane mode, fullscreen kiosk visualizer mode, and controller-backed reopening.

Push to moOde behavior

Direct file inspection shows a concrete pushToMoode() flow:

  1. fetch /config/runtime
  2. extract the track key
  3. build a /visualizer.html?kiosk=1... URL
  4. POST visualizer state to /peppy/last-profile
  5. POST the browser target URL to /config/moode/browser-url

That ties visualizer designer state into live moOde browser-target updates.

Relevant API calls

  • Read: GET /config/runtime, GET /peppy/last-profile
  • Write: POST /peppy/last-profile, POST /config/moode/browser-url

Special embedded-mode rendering differences

Embedded mode appears to affect more than chrome. Direct file clues suggest some scene/timing behavior also changes in embedded mode (for example an aeon intro lead-in path), so embedded mode is best understood as the same visual engine under stricter presentation constraints.

What this page is now confident about

  • visualizer.html is fundamentally a multi-scene reactive visual engine
  • it uses music-driven feature state such as volume/loudness/spectrum/event signals to drive animation
  • it has a substantial preset/configuration layer
  • embedded mode is a hosting mode that hides controls and emphasizes pure visual presentation
  • fullscreen kiosk mode and embedded pane mode are navigation contexts around the same underlying visual system
  • the page participates in live moOde push behavior via runtime config, peppy/last-profile, and browser-target update flow

Current status

The strongest current truth is that visualizer.html is first a reactive visual engine with multiple models/scenes that respond to live music-driven data, and only secondarily a child page that can run in embedded pane mode.
