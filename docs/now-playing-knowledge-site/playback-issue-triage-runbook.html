playback issue triage runbook

Purpose

This runbook gives a practical triage order for playback-side problems in now-playing.

It is meant for issues like wrong current song, wrong art, wrong metadata, wrong status truth, queue head / Next Up disagreement, mode-specific behavior mismatches, and playback symptoms caused by route mediation, fallback logic, or host/runtime reality.

Core rule

When a playback issue appears, triage in this order:

  1. classify the playback mode
  2. identify the likely authority chain
  3. decide whether the symptom belongs to transport, queue, art/metadata, or presentation interpretation
  4. check historically fragile playback paths
  5. verify on the layer that should actually own the truth

Step 1: classify the playback mode first

  • Local file playback
  • AirPlay
  • UPnP
  • Radio / stream
  • Unresolved / fallback-driven case

Important guardrail: if the mode is misclassified, the rest of the debugging path is usually wrong.

Step 2: choose the matching authority chain

  • Local file: trust MPD playback/queue truth, local file/library identity, and moOde/local-file art path.
  • AirPlay: trust moOde current-song/status as interpreted for AirPlay, aplmeta.txt, and AirPlay-specific art/metadata handling.
  • UPnP: trust moOde current-song/status, UPnP-specific resolution logic, and any recovered local-library truth.
  • Radio/stream: trust the interpretation path, not raw metadata alone: holdback, cleanup, enrichment, guardrails, and logo/fallback behavior.
  • Fallback-driven: trust last-good behavior, cache-backed art/current-state fallback, and minimal surviving authoritative state.

Step 3: decide what kind of playback symptom this is

  • Transport truth problem
  • Queue truth problem
  • Art / metadata truth problem
  • Presentation interpretation problem

Important guardrail: do not treat every playback symptom as a low-level transport bug.

Step 4: check the historically fragile playback paths

  • Next Up / Alexa-mode behavior
  • Mode-sensitive art/current-song/status truth
  • Radio metadata guardrails
  • Last-good / fallback-preservation behavior
  • AirPlay local override reality

Each of these has a different recommended starting point, but the shared rule is to widen early when the symptom touches a known fragile path.

Step 5: decide which layer to inspect first

  • Route/bootstrap layer: moode-nowplaying-api.mjs, art.routes.mjs, queue.routes.mjs, diagnostics routes
  • Service layer: src/services/mpd.service.mjs
  • Page/runtime layer: top-level page entrypoint, scripts/index-ui.js, page runtime logic
  • Local runtime/override layer: runtime-admin routes, local environment, moOde override material

Step 6: use the minimum verification set that matches the symptom

  • Transport/control wrong: verify route/diagnostics behavior, actual playback state, and one affected surface
  • Queue head / Next Up wrong: verify queue/head truth, /next-up, one controller surface, and Alexa-mode differences if relevant
  • Art/metadata wrong: verify playback mode, art/current route truth, fallback/guardrail behavior, and one visible surface
  • Degraded or stale-looking output: verify whether last-good/fallback behavior is intentionally active and whether the output is degraded-but-correct

Step 7: declare success only when the right truth source and visible result agree

A playback issue is solved when the playback mode is correctly classified, the correct authority chain is identified, the owning route/service layer behaves correctly, and the visible surface reflects that truth correctly.
