agent start

Purpose

This page is a question-first entry point for agents using the rendered now-playing wiki.

Its job is not to explain the whole system. Its job is to get you to the right page quickly, with less wandering and fewer wrong first guesses.

How to use this page

  1. Classify the task by question type.
  2. Open the recommended starting page.
  3. Use quick search to narrow by concept or heading.
  4. Only then drop into lower-level drill-downs, anatomy pages, or route ownership pages.

Question-first entry points

What is this system and how is it shaped?

  • Start with system-overview
  • Then architecture
  • Then user-interfaces

Where do I start looking in code?

  • Start with source-map
  • Then route-ownership-map
  • If route/API-centric, also open api-service-overview

Which page or module likely owns this behavior?

  • Start with route-ownership-map
  • Then fragile-behavior-ownership if the behavior is historically brittle
  • Then the most relevant surface/anatomy page

Is this queue, playback, or queue shaping?

  • Start with queue-and-playback-model
  • Then controller-queue-interface
  • Then playback-authority-by-mode if source-of-truth is unclear

What is the source of truth for this visible state?

  • Start with playback-authority-by-mode
  • Then route-ownership-map
  • Then fragile-behavior-ownership

I need the operator/runtime truth surface

  • Start with config-interface
  • Then diagnostics-interface
  • Use local-environment if live behavior may be host-specific

I need the specific card, region, or module inside a page

  • Start with the relevant anatomy page if it exists
  • Examples: config-page-anatomy, diagnostics-page-anatomy, library-health-page-anatomy, theme-page-anatomy, kiosk-shell-anatomy

This looks like display or kiosk behavior

  • Start with display-interface
  • Then kiosk-interface if shell/presentation mode is involved
  • Then the display/kiosk troubleshooting or anatomy companions as needed

This looks like metadata, library, or artwork cleanup

  • Start with library-health-interface
  • Then library-health-page-anatomy

This looks like theme, token, or preset behavior

  • Start with theme-interface
  • Then theme-page-anatomy

High-value do-not-confuse-with

  • Queue interface is not the same as Queue Wizard.
  • Kiosk is a presentation/shell mode, not just another render mode.
  • Theme is a client-side token editor, not a normal runtime config page.
  • Config is an operator console, not just a settings form.
  • Library Health is a maintenance workbench, not just a report page.

Confidence and proof heuristics

When reading the wiki, treat pages roughly like this:

  • Higher confidence: route ownership pages, anatomy pages, pages backed by direct repo inspection
  • Medium confidence: structural hub pages and branch summaries
  • Lower confidence / more provisional: pages that explicitly say proof is still incomplete or route inventory is still open

If a page says “still to verify,” believe it.
