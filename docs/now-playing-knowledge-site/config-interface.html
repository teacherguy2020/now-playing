config interface

Purpose

This page documents now-playing/config.html, the main operator-facing configuration console in the now-playing ecosystem.

This page is intentionally implementation-aware, because config.html is clearly not just a static settings form. It acts as a broad operational console for reading runtime config, editing and saving configuration, verifying environment assumptions, driving maintenance actions, and checking integrations and service health.

Why this page matters

config.html appears to be one of the central operator surfaces in the project. It combines configuration editing, environment validation, service/integration maintenance actions, operator feedback and restart flow, and multiple feature-specific maintenance tools in one place.

High-level role

A good current interpretation is that config.html is the main broad-spectrum operator console. It is both a configuration editor and an operational maintenance surface, and it mediates between human-entered settings and runtime/API-backed verification flows.

Shell / entry behavior

config.html contains shell-redirect behavior near the top of the page. If standalone=1, it remains in standalone mode; if loaded top-level without that flag, it redirects into app.html?page=config.html; if already embedded in a shell, it does not redirect.

Major UI areas visible in the page

Repo-visible structure shows that config.html includes network & runtime configuration, podcasts controls, mpdscribble controls, Alexa configuration, ratings DB maintenance, full raw JSON editing, restart-services modal flow, and pre-hero status pills for API/Web/Alexa/moOde visibility.

Key DOM/action centers

Important operator-facing elements include #checkRuntimeBtn, #saveBtnTop, #saveBtn, #saveFullBtn, #reloadBtn, #modalRestartBtn, mpdscribble controls, Alexa checks, ratings DB controls, and JSON formatting controls. This is action-heavy, not just field-heavy.

Runtime config as the center of gravity

One of the clearest architectural facts in config.html is that the page revolves around GET /config/runtime and POST /config/runtime. These appear to be the core read/write endpoints for the page.

Key functions / logic blocks

loadCfg()

The main config-loading function, responsible for fetching runtime config, hydrating UI fields, cached track-key state, feature toggles, correction tables, and the full raw JSON editor.

saveCfg()

The main structured save path, validating track key and required host values, gathering structured config, POSTing to /config/runtime, and updating saved-state baseline.

saveFullCfg()

The raw full-config save path, reading JSON from the editor, validating/parsing it, and POSTing to /config/runtime.

saveArtistCorrectionsOnly()

A narrower targeted save path for artist/album/playlist correction structures.

checkRuntimeEnv()

One of the most important operational functions: it requires a track key, gathers MPD/SSH/path/runtime values, POSTs to /config/runtime/check-env, updates runtime status UI, and unlocks or blocks downstream config cards based on result.

ensurePodcastRoot()

An environment-fixing action that POSTs to /config/runtime/ensure-podcast-root.

restartServicesNow()

Connected to the save/restart modal flow and POSTs to /config/restart-services.

Important API calls

Important API endpoints used by config.html include GET /config/runtime, POST /config/runtime, POST /config/runtime/check-env, POST /config/runtime/ensure-podcast-root, POST /config/restart-services, GET /config/services/mpdscribble/status, POST /config/services/mpdscribble/action, POST /config/alexa/check-domain, and ratings maintenance routes.

Track key handling

config.html places strong emphasis on the track key. It is cached locally, synced into visible fields, required before many runtime/save actions, and used in protected maintenance requests. It is one of the central operator credentials for the page’s write capabilities.

Alias and correction management

The page also includes substantial correction-management UI for artist aliases, album aliases, playlist aliases, and unresolved heard values and promotion/dismiss flows.

Save / reload / restart model

A useful current model for the page’s operator workflow is: load runtime config, edit structured fields or raw JSON, optionally run environment verification, save config via structured or raw path, and optionally restart services through the modal flow.

Architectural interpretation

A good current interpretation is that config.html is the broad operational front-end for runtime config, bundles multiple admin and maintenance sub-workflows, and is one of the strongest bridges between UI, runtime API, and live environment assumptions.
