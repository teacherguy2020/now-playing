source map

Purpose

This page is the navigation map for the now-playing codebase.

It should help future agents and humans answer questions like where to look first, what directory probably owns a behavior, and what parts are runtime/integration-heavy versus UI-heavy.

High-level repository areas

moode-nowplaying-api.mjs

Current main server/bootstrap assembly point for the API.

Related API branch pages:

  • api-service-overview.md
  • api-config-and-runtime-endpoints.md
  • api-youtube-radio-and-integration-endpoints.md
  • api-playback-and-queue-endpoints.md

Practical implication: for real API assembly/runtime wiring questions, start here before assuming a root index.js-style entrypoint owns the server.

src/

Primary application source tree with important subareas including src/config/, src/lib/, src/routes/, src/services/, and src/private/.

Observed refinement: src/lib/ looks more like a shared helper/data layer, while src/routes/ contains many focused route modules assembled by the main server bootstrap.

scripts/

Operational/build/helper scripts and one-off automation support.

styles/, assets/, skins/

Presentation, visual assets, and skin/theme-specific material.

integrations/

Integration-specific code and support files, especially integrations/moode/.

ops/

Operational and environment-specific material, especially ops/moode-overrides/ as a recovery/audit mirror of local moOde overrides.

config/ and src/config/

Top-level configuration artifacts plus code-facing configuration models/loaders.

alexa/, lambda_bundle/, lambda_upload/

Alexa/Lambda-related integration and deployment material.

First places to look by task

UI or visual behavior

Start with the actual user-facing surface first:

  • Desktop app shell → app.html
  • Tablet controller → controller-tablet.html
  • Phone controller → controller-mobile.html
  • moOde box display surfaces → kiosk.html, peppy.html, player.html / player-render.html
  • display/router/classic now-playing layer → display.html, index.html

Within that family, “Peppy skin” is useful shorthand for a visual/theme variant inside the broader Peppy display mode.

Then check src/routes/, styles/, skins/, and assets/.

Playback or control behavior

Start with src/services/, src/routes/, and src/lib/.

Related pages: api-playback-and-queue-endpoints.md, route-ownership-map.md, playback-mode-troubleshooting.md.

Integration behavior

Start with integrations/, then src/services/ / src/routes/, and only then ops/ when local overrides may explain the behavior.

Deployment/runtime issues

Start with ops/, scripts/, config/, src/config/, plus project docs and local environment notes.

Alexa or voice-related work

Start with alexa/, lambda_bundle/, and lambda_upload/.

Observed architectural patterns

  • The repo is not cleanly organized as “thin routes, fat services.” Many important behaviors are route-heavy.
  • Top-level HTML pages are first-class ownership points for user-facing surfaces.
  • src/routes/ often owns substantial operational workflows directly.
  • src/services/ contains real reusable primitives, but selectively.
  • src/lib/ is a shared helper/data layer, not a dumping ground.
  • Runtime-admin routes are especially important because they bridge config mutation, environment discovery, and SSH-backed moOde/display side effects.
