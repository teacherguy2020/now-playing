controller queue interface

Purpose

This page documents controller-queue.html as the queue interface surface in the now-playing system.

It makes explicit what earlier pages only implied: the queue page is a real standalone and embedded controller surface.

Important scope note

Current wiki work already supports that controller-queue.html is a real queue UI surface, supports embedded=1, appears inside substantive now-playing surfaces, and participates in kiosk right-pane routing. What is not yet fully proven is the exact backend route inventory and owning file/module chain.

High-level role

A good current interpretation is:

  • controller-queue.html is the main queue inspection/control surface
  • it is where the user/operator can view the current queue in a dedicated UI
  • it is also the reusable queue UI that other surfaces embed when they need queue access without leaving the current page

Queue interface versus Queue Wizard

controller-queue.html is primarily about queue visibility and direct queue interaction. Queue Wizard is the higher-order queue-shaping and preview/apply layer. Related, but not identical.

Evidence from current surface work

  • Substantive controller-now-playing* pages embed controller-queue.html and controller-queue.html?embedded=1 in queue modals.
  • controller-queue.html participates in the embedded-pane contract via embedded=1 and theme/color/layout propagation.
  • Kiosk routing maps kiosk-queue.html to controller-queue.html in the right pane.

Standalone versus embedded role

Standalone mode

In standalone mode, controller-queue.html is the dedicated queue page with queue state as its primary responsibility.

Embedded mode

In embedded mode, controller-queue.html?embedded=1 acts as a reusable queue subview inside now-playing modals and kiosk panes, compacting layout and cooperating with parent-shell expectations.

Queue interface in the broader queue/playback model

Using the established layered model:

  • Layer 1 = transport
  • Layer 2 = raw queue mutation
  • Layer 3 = queue inspection / direct queue interface
  • Layer 4 = queue shaping / curation (Queue Wizard)

controller-queue.html is the clearest Layer 3 surface currently identified.

Relationship to now-playing surfaces

The queue interface is tightly tied to the controller-now-playing* family. Queue UI issues may actually belong to controller-queue.html, its embedded-mode behavior, or its backend queue contract rather than to the top-level now-playing surface itself.

Relationship to kiosk mode

Because kiosk right-pane routing maps kiosk-queue.html to controller-queue.html, queue access in kiosk mode is controller-backed rather than a kiosk-only duplicate.

For a faster answer to the recurring question of shell owner versus content owner, see kiosk-queue-shell-vs-content-owner.

Relationship to backend/API proofing

The queue interface surface is already clear, but deeper proof is still needed for which routes populate the page, which routes mutate queue state from inside it, and how much is MPD-direct versus app-mediated.

Current status

This page gives the wiki a needed missing anchor in the playback/queue branch: controller-queue.html is the queue interface surface, it has standalone and embedded roles, it is used by now-playing and kiosk flows, it is distinct from Queue Wizard, and its backend contract still needs deeper exact proof later.
