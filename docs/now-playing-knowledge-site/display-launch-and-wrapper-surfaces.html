display launch and wrapper surfaces

Purpose

This page documents the display-oriented helper pages that launch, wrap, or redirect display surfaces instead of rendering them directly.

Current pages: displays.html, controller-visualizer.html, and index1080.html.

displays.html — display target launcher and moOde push console

Direct file inspection shows that displays.html is a real operator console, not just a menu. It presents two groups of cards:

  • Now Playing Displays: Player, Peppy, Visualizer, Index Display
  • Control Displays: Mobile Controller, Kiosk Controller 1280×400

Open-designer links go to:

  • app.html?page=player.html
  • app.html?page=peppy.html
  • app.html?page=visualizer.html&ui=1
  • index.html
  • app.html?page=mobile.html
  • app.html?page=kiosk-designer.html

The page computes an app-host base URL at :3101, fetches /config/runtime to obtain config.trackKey, and uses that key for privileged POSTs.

The core helper is pushUrl(url, reason, ...), which POSTs to /config/moode/browser-url with JSON { url, reason }.

For Player, Peppy, and Visualizer, the page first POSTs to /peppy/last-profile and then pushes the same browser target:

  • http(s)://<host>:8101/display.html?kiosk=1

The selected mode is carried in the /peppy/last-profile payload, not by unique pushed URLs.

Observed mode payloads include:

  • Player → { url, displayMode: 'player', playerSize }
  • Peppy → { url, displayMode: 'peppy' }
  • Visualizer → { url, displayMode: 'visualizer' }

Player push also reads localStorage['player.profile.v2'] to extract a saved size, defaulting to 1280x400.

Kiosk behaves differently: it pushes http(s)://<host>:8101/kiosk.html directly with reason displays-kiosk-push and does not update /peppy/last-profile.

Index Display only offers Open Designer to index.html; there is no push button in the current file.

Best interpretation: displays.html is the operator-facing bridge from selection UI into live moOde browser-target updates.

controller-visualizer.html — thin wrapper host for visualizer.html

This page is a small shell around an iframe, with a top bar and back button in standalone mode and a full-bleed embedded mode when embedded is truthy.

It builds a child URL for visualizer.html, passing through only a curated set of parameters:

  • preset
  • energy
  • motion
  • glow
  • ui
  • kiosk
  • theme
  • colorPreset

If absent, it forces:

  • ui=0
  • kiosk=1

Then it loads the normalized child URL into #vizFrame and uses the back button to return to controller.html.

Best interpretation: controller-visualizer.html is a controller-friendly host/adaptor for visualizer.html, not the visualizer implementation itself.

index1080.html — redirect shim

Direct file inspection shows only:

  • a meta refresh to /index.html
  • a fallback body link to /index.html

There is no scripting or wrapper logic. This is just a legacy/compatibility redirect shim.

Key distinctions to preserve

  • displays.html vs display.html: launcher/push console vs actual shared display target entrypoint
  • controller-visualizer.html vs visualizer.html: host shell vs actual visualizer implementation
  • index1080.html vs index.html: redirect shim vs real page
