architecture

Purpose

This page describes the now-playing system in terms of major areas and relationships.

It is more structural than system-overview.md: less about what the project is broadly for, more about how the major pieces relate, while still staying above a file-by-file implementation map.

Major architectural areas

  • Media and content layer — media library, playlist management, radio stations, ratings/favorites
  • Playback and control layer — transport controls, queue wizard, filter builder, quick search flows, controller surfaces, and device-specific aliases
  • Display and presentation layer — display pages, kiosk mode, animated art, art caching, moOde display control
  • Local derived-state and cache layer — localized library/browse index JSON, cached/resized/blurred art derivatives, and other stabilizing artifacts
  • Integration layer — MPD, YouTube, Alexa, radio metadata evaluation, HTTP/API request patterns
  • Operational layer — installation/setup, configuration management, service control, runtime verification, host/runtime checks

Relationships between areas

These areas are not independent. Media/content supplies what can be played and displayed, control shapes playback behavior, display exposes state and visual output, derived-state/cache bridges live systems to faster or more stable local representations, integration connects internal behavior to outside systems, and the operational layer keeps all of that functioning in a live environment.

Cross-layer interactions that should stay explicit:

  • control actions can immediately affect display state
  • playback behavior can be constrained by integration or runtime state
  • integration behavior can be shaped by local environment realities rather than only project code
  • operational/runtime conditions can invalidate assumptions made by higher layers

Repo-verified refinement:

  • many important behaviors sit at the intersection of UI, playback, integration, and runtime state
  • the codebase is not cleanly organized as “thin routes, fat services”
  • top-level HTML entrypoints are first-class architectural ownership points
  • thin redirect/alias HTML pages still matter architecturally when they normalize callers onto real controller shells
  • helper/operator pages like displays.html, radio-eval.html, youtube.html, and alexa.html are significant architectural surfaces

Display, control, and playback layers

These three layers are especially tightly related. Changes that look like display issues may still depend on control logic or playback/integration state; changes that look like queue/control issues may still affect visible presentation; and changes that look frontend-only may still depend on route-local workflows, runtime-admin behavior, or moOde/display control side effects.

Integration boundaries

The docs make several boundaries visible: the internal system boundary where display/control/media features interact, the external integration boundary where the project interacts with MPD/YouTube/moOde and request flows, and the runtime/host boundary where the project depends on the live environment.

Repo-verified refinement includes route modules owning substantial operational behavior directly, services existing more selectively as reusable primitives, shared helper/data layers such as browse-index and art caches, and the practical search architecture depending on localized browse-index data rather than only live upstream state.

Operational dependencies

The project clearly depends on operational context including runtime tools and setup, service/process health, current runtime endpoints/URLs, display/wake/runtime configuration, and local-environment-specific patches and overrides.

Brian’s local environment is part of the real operating architecture, not just a deployment afterthought.

Architectural cautions

  • do not treat this project as a purely frontend app
  • do not assume display behavior is independent from runtime or integration behavior
  • do not assume documented project-wide behavior always matches the local installation
  • check local-environment.md and deployment-and-ops.md when architecture questions involve live behavior
  • use source-map.md when moving from architectural understanding into actual code changes

Architectural questions future agents should ask first

  • is this problem primarily in media/content, control, display, integration, or runtime/ops?
  • is this behavior project-wide, or is it likely shaped by Brian’s specific local environment?
  • does this issue cross a layer boundary?
  • would fixing this require only code changes, or also runtime verification and environment awareness?
