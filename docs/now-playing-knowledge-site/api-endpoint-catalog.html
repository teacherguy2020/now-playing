api endpoint catalog

Purpose

This page is the first source-driven endpoint catalog for the now-playing app-host API.

It is based directly on source-visible routes, mainly in moode-nowplaying-api.mjs and src/routes/*.mjs.

Scope and proof model

This page is strong on endpoint existence and owner-file visibility, but not yet a full request/response contract spec for every route.

How to use this page

Use this page when the question is whether an endpoint exists, what family it belongs to, or which source file likely owns it. Also note that diagnostics.html has an endpoint-search/request-runner surface backed by scripts/diagnostics.js, including a server-loaded list from GET /config/diagnostics/endpoints and a built-in fallback catalog.

  • POST /config/queue-wizard/vibe-start

Observed request-shape examples:

  • POST /config/queue-wizard/preview → body includes genres, artists, albums, excludeGenres, minRating, maxTracks, and related preview flags like varietyMode.
  • POST /config/queue-wizard/apply → body includes mode, keepNowPlaying, tracks, shuffle, plus related flags like saveOnly, forceRandomOff, fastStart, and generateCollage.
  • POST /config/queue-wizard/vibe-start → diagnostics example includes targetQueue and minRating.
  • POST /config/queue-wizard/vibe-seed-start → diagnostics example includes seedArtist, seedTitle, and targetQueue.

Diagnostics / debug

  • POST /config/browse-index/rebuild
  • GET /config/diagnostics/endpoints
  • GET /config/diagnostics/album-tracks
  • GET /config/diagnostics/artist-albums
  • POST /config/diagnostics/queue/save-playlist
  • POST /config/diagnostics/playback
  • GET /config/diagnostics/queue
  • GET /debug/radio-metadata-log
  • POST /debug/radio-metadata-log/clear

The diagnostics request runner uses GET /config/diagnostics/endpoints as a server-backed endpoint list and falls back to a built-in endpoint catalog in scripts/diagnostics.js. The route itself merges a curated endpoint catalog with discovered app routes and exposes metadata like group/name/body/query hints.

Observed request-shape examples:

  • POST /config/diagnostics/playback → diagnostics example body includes action: 'play'.
  • POST /config/diagnostics/queue/save-playlist → diagnostics example body includes playlistName and generateCollage.
  • GET /config/diagnostics/album-tracks → query includes album and artist.
  • GET /config/diagnostics/artist-albums → query includes artist.

Config runtime / services / moOde control

  • GET/POST /config/runtime

  • GET /config/services/mpdscribble/status
  • POST /config/services/mpdscribble/action
  • POST /config/alexa/check-domain
  • GET/POST /config/controller-profile

Observed request/response-shape examples:

  • GET /config/runtime returns fields including ok, configPath, config, and fullConfig.
  • POST /config/runtime/check-env takes body fields like mpdHost, mpdPort, sshHost, sshUser, and paths, and returns fields like sshOk, mpdOk, sshError, mpdError, and pathChecks.
  • POST /config/runtime/ensure-podcast-root takes sshHost, sshUser, and podcastRoot, and returns ok, podcastRoot, and created.
  • POST /config/services/mpdscribble/action uses an action field such as restart.

YouTube

  • POST /youtube/resolve
  • POST /youtube/search
  • POST /youtube/playlist
  • GET /youtube/proxy/:id
  • POST /youtube/queue

Observed request-shape examples:

  • POST /youtube/resolve → body field url; returns success with resolved payload or 400 if missing.
  • POST /youtube/search → body fields query, limit, and playlistsOnly.
  • POST /youtube/playlist → body fields include url and limit.
  • POST /youtube/queue → body fields include mode plus url or urls[]; mode supports append, replace, and crop.
