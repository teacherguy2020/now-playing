api playback and queue endpoints

Purpose

This page documents the playback- and queue-oriented endpoint families that are already visible through current wiki work.

It is intentionally careful: it covers the route families we already have evidence for without pretending we already have a full exhaustive queue-route inventory.

Why this page matters

Playback and queue control are some of the most important practical behaviors in the project. They are where the system turns user actions into play/pause/skip behavior, queue mutation, album/playlist insertion, kiosk/controller control changes, and display-side push behavior that can affect active playback/presentation workflows.

Auth / request model

  • HTTP GET and POST are both used
  • some families are explicitly track-key-protected
  • queue- and playback-affecting behavior often sits close to controller/UI actions rather than behind a deeply abstracted backend model

1. Direct playback control family

Current wiki work already shows direct playback control calls such as /config/diagnostics/playback.

A good current interpretation is that this route family accepts direct transport-style playback commands from controller/now-playing surfaces and is part of the app-host control plane for transport behavior.

2. Queue-wizard preview/apply family

Current wiki work repeatedly references queue-wizard preview/apply flows for album-level and related queue-shaping actions.

A good current interpretation is that there is a queue-wizard route family that supports previewing and applying queue changes before committing them.

Important note: the exact route names are not yet fully enumerated here, so this page keeps that claim at the route-family level.

3. YouTube queue submission family

Current wiki work already shows one concrete queue-affecting integration family: POST /youtube/queue.

Documented request shape already includes fields like url, urls, and mode, with queue modes including append, crop, and replace.

4. Embedded queue loading patterns

Current wiki work shows recurring controller-side queue behavior such as queue modals with embedded controller-queue.html and controller-queue.html?embedded=1.

This tells us queue inspection and queue control are first-class controller behaviors even though the exact backend contract behind that queue page still needs deeper inventory work.

5. Display-push routes that overlap playback/presentation control

Current wiki work on displays.html already shows POST /config/moode/browser-url and POST /peppy/last-profile.

These are not pure playback routes, but they sit close enough to the operational playback/presentation boundary to remain adjacent to this branch.

Relationship between these families

  • direct transport routes handle immediate playback actions
  • queue-wizard routes handle richer queue shaping and preview/apply flows
  • integration-specific queue routes like /youtube/queue feed content into the queue subsystem
  • controller queue pages are UI shells over a deeper queue API contract not yet fully inventoried here
  • some display-push routes overlap playback/presentation control enough to remain nearby

What this page is not yet

This page is not yet the final exhaustive queue API inventory. It still does not fully enumerate every queue route, every playback transport action name, every queue-wizard route name, or every request/response shape used by controller queue pages.
