api playback and queue endpoints

Purpose

This page documents the playback- and queue-oriented endpoint families that are already visible through current wiki work.

It is intentionally careful: it covers the route families we already have evidence for, without pretending we already have a full queue-route inventory.

1. Direct playback control family

Current wiki work already shows direct playback control calls such as:

  • /config/diagnostics/playback

This family appears in controller-now-playing surfaces and is part of the app-host control plane for transport behavior.

2. Queue-wizard preview/apply family

Current wiki work repeatedly references queue-wizard preview/apply flows for album-level and related queue-shaping actions.

The exact route names are not yet fully enumerated, but the route family clearly exists and is part of richer queue curation behavior.

3. YouTube queue submission family

Current wiki work already shows one concrete queue-affecting integration endpoint:

  • POST /youtube/queue

with queue modes including:

  • append
  • crop
  • replace

So even though it lives in the YouTube branch, it also belongs in playback/queue coverage.

4. Embedded queue loading patterns

Current controller/now-playing pages repeatedly load:

  • controller-queue.html
  • controller-queue.html?embedded=1

This tells us queue inspection/control is first-class controller behavior, even though the exact backend route list behind that queue page still needs deeper code-first inventory work.

5. Display-push routes that overlap playback/presentation control

Current wiki work on displays.html already shows:

  • POST /config/moode/browser-url
  • POST /peppy/last-profile

These are not pure playback routes, but they sit close enough to the operational playback/presentation boundary that they belong adjacent to this branch.

Current status

This page gives the API branch a credible playback/queue overview while explicitly calling out where deeper queue-route inventory still needs to happen later.
