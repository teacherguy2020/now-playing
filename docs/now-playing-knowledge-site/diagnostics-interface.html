diagnostics interface

Purpose

This page documents now-playing/diagnostics.html, the diagnostics and live-inspection console in the now-playing ecosystem.

This page is intentionally implementation-aware, because diagnostics.html is clearly not just a passive status screen. It acts as a multi-purpose operator console for making direct API requests, inspecting responses, testing endpoints with or without x-track-key, viewing live embedded UI surfaces side-by-side, and navigating operational/debug workflows through a browser UI.

Why this page matters

diagnostics.html appears to be one of the strongest operator-facing truth surfaces in the project. It combines direct endpoint execution, live embedded views of multiple UI surfaces, clipboard/curl-oriented debugging helpers, and quick comparison between API behavior and rendered UI behavior.

Important files

  • Primary page: now-playing/diagnostics.html
  • Important supporting script: now-playing/scripts/diagnostics.js

Related pages include configuration-and-diagnostics-interfaces.md, config-interface.md, display-surface-troubleshooting.md, and deployment-and-ops.md.

High-level role

A good current interpretation is that diagnostics.html is a browser-based operator/debug console, helps bridge raw API behavior and visible UI behavior, and is one of the most practical pages for side-by-side inspection and debugging.

Shell / entry behavior

Like config.html, diagnostics.html contains shell-redirect behavior: if standalone=1, remain standalone; if top-level without standalone, redirect into app.html?page=diagnostics.html; if embedded in another shell already, do not redirect.

Major UI areas visible in the page

1. Request runner / endpoint console

This top card includes endpoint filtering/selection, method selection, path field, JSON body field for POST, run button, favorite toggle, copy-as-curl button, copy-response button, optional send x-track-key checkbox, and a status area.

2. Live embedded view cards

The page includes multiple live embedded-view cards for direct UI inspection, including live now-playing view, live player view, live peppy skin view, live controller view, and live kiosk view.

Important DOM/action centers

Observed important elements include request-runner controls like #endpointFilter, #endpoint, #method, #path, #body, #runBtn, #favBtn, #copyCurlBtn, #useTrackKey, #copyBtn, and #status, plus live-view frames and their reload/toggle/zoom controls.

Diagnostics script as the logic center

The main page includes scripts/diagnostics.js, which strongly suggests most of the request-running, favorites, frame-management, and interaction logic lives there rather than inline.

Anatomy companion page

diagnostics-page-anatomy.md

This is the anatomy-style companion page for diagnostics.html. Use it when the task is not just about Diagnostics as a surface, but about a specific region such as the request runner, favorites/endpoint selector, track-key toggle, curl/response helper layer, or embedded live-surface cards.

Request-runner behavior

Based on visible structure, the request runner supports selecting or filtering known endpoints, choosing method, editing path manually, submitting JSON for POST requests, optionally attaching x-track-key, copying equivalent curl, and favoriting endpoints.

Track-key-aware diagnostics

The diagnostics page includes #useTrackKey, implying the console is designed to test both open/unprotected behavior and protected/track-key-backed behavior.

Live embedded-surface inspection

This is one of the strongest and most distinctive aspects of diagnostics.html. The page supports a workflow like: run or inspect an endpoint, observe the live rendered result in one or more surfaces, compare whether the API and the UI agree, then reload or reopen affected views quickly.

Relationship to display and controller debugging

Because diagnostics embeds multiple live UI surfaces, it sits at the junction of config/diagnostics/admin, display surfaces, controller surfaces, kiosk behavior, and peppy/player render behavior.

Current observed API/config contact

Even from the current light inspection, we can already see that diagnostics.html fetches runtime config from /config/runtime, conditionally hides/shows some navigation based on runtime config, and is designed to execute arbitrary configured diagnostic requests through the request-runner UI.

Architectural interpretation

A good current interpretation is that diagnostics.html is the browser operator’s live debugging dashboard. It combines API-console behavior with live surface monitoring and is one of the most direct pages for checking whether data flow and rendered UI are aligned.

Things still to verify

Future deeper verification should clarify the exact endpoint catalog and grouping inside scripts/diagnostics.js, how favorites are stored and restored, how curl-copy generation is implemented, whether the request runner can safely target all important protected endpoints, how the live embedded cards are initialized and refreshed, and whether diagnostics includes hidden/advanced features not visible from markup alone.

Current status

At the moment, this page already has enough evidence to be treated as one of the project’s core operator/debugging surfaces. It is not just a diagnostics readout; it is a live API console plus multi-surface inspection dashboard, which makes it especially valuable for real troubleshooting work.
