kiosk shell anatomy

Purpose

This page documents the anatomy of the live kiosk shell path in the now-playing project.

It distinguishes between the kiosk launcher/bridge, the live kiosk shell behavior, the right-pane embedded child region, the kiosk designer/push tool, and the separate controller-kiosk.html scaffold path.

The most important structural fact

kiosk.html is mainly a launcher/profile bridge. It does not appear to be the final live kiosk UI. It redirects into controller.html with kiosk-flavored query params.

So the live kiosk shell path is best understood as:

  1. kiosk.html
  2. local profile/query-param merge
  3. redirect into controller.html?kiosk=1&preview=1&...
  4. controller-backed kiosk shell behavior
  5. optional right-pane embedded child pages

Main anatomical regions of the kiosk shell path

  1. launcher/profile bridge (kiosk.html)
  2. controller-backed live kiosk shell
  3. right-pane embedded child region
  4. kiosk-branded child-route aliases
  5. kiosk designer / preview / push tool
  6. separate scaffold/prototype path (controller-kiosk.html)
  7. runtime/display-host-sensitive push and target behavior

Launcher/profile bridge

kiosk.html reads query params and localStorage profile state, synchronizes a controller/mobile profile, and redirects into controller.html with kiosk-related query params.

Controller-backed live kiosk shell

The actual live kiosk shell appears to be controller-backed after the redirect from kiosk.html. This is where kiosk-aware layout, theme/color application, right-pane behavior, and embedded child hosting likely live.

Right-pane embedded child region

This region hosts child pages such as now-playing, albums, artists, playlists, radio, podcasts, genres, queue, and queue wizard inside the live kiosk shell.

Kiosk-branded child-route aliases

Many kiosk-*.html pages are kiosk-branded redirect aliases into controller-family pages such as controller-now-playing.html, controller-albums.html, and controller-queue.html.

Kiosk designer / preview / push tool

kiosk-designer.html is the authoring/preview/push tool. It provides iframe preview of kiosk.html, manages color/preset/custom-theme state, persists kiosk profile state to localStorage, supports export/import, and includes Push to moOde behavior.

Separate scaffold/prototype path

controller-kiosk.html is a separate kiosk/controller scaffold path: fixed 1280×400 layout, split panes, direct app-host fetches. It should not be assumed to be the same as the main live kiosk shell path.

Runtime/display-host-sensitive behavior

Kiosk behavior depends strongly on runtime/display-host-sensitive control paths such as Push to moOde, browser target URL checks, app-host runtime/admin routes, moOde-host actual display behavior, and local overrides.

Practical “where do I start?” map

  • Kiosk launch/profile behavior: start with kiosk.html.
  • Live kiosk shell: start with controller-backed kiosk shell behavior and controller-kiosk-mode.md.
  • Right pane: start with kiosk-right-pane-routing.md, embedded-pane-contracts.md, and the actual child page.
  • Preview/push tooling: start with kiosk-designer.html.
  • 1280×400 scaffold: start with controller-kiosk.html.
  • Wrong thing on real display: start with display runbooks, runtime-admin/display-target behavior, and local environment reality.

Current status

This is the fourth real page-anatomy reference in the wiki. Its main job is making explicit that “kiosk” is not a single page but a multi-part shell path.
