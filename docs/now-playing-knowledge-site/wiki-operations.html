wiki operations

Purpose

This page defines the operating model for maintaining the now-playing knowledge wiki.

A useful current model is: ingest, query, and lint.

Why this page matters

The wiki is most useful when it compounds. That means new evidence gets integrated into existing pages, useful answers become durable pages instead of disappearing into chat, and stale/confusing/orphaned structure gets corrected periodically.

Three-layer mental model

  • raw source layer — repo files, runtime observations, local environment facts, direct code tracing
  • wiki layer — the maintained markdown and rendered pages that synthesize and organize those facts
  • schema/rules layer — the instructions that define how the wiki should be updated and used

1. Ingest

Ingest means taking some new source of truth and integrating it into the wiki.

Typical inputs include repo inspection, code tracing, runtime observations, deployment findings, user decisions about framing, and recurring answers discovered during problem-solving.

Ingest should usually produce one or more of:

  • a new page
  • a meaningful update to an existing page
  • a new companion/anatomy/synthesis page
  • better cross-links
  • a rendered-page update

Rule: do not create new pages as filler. A source should improve confidence, navigability, ownership clarity, or question-answering usefulness.

2. Query

Query means using the wiki to answer a practical question.

The normal pattern is:

  1. start from a hub, search page, or agent-start page
  2. move into the relevant branch page(s)
  3. move into anatomy/ownership/synthesis pages
  4. only then drop to raw code or runtime verification

Important: if a query reveals a recurring confusion, the answer should often be filed back into the wiki as a durable page.

3. Lint

Lint means checking the health of the wiki itself.

A useful lint pass should look for:

  • orphan pages
  • missing parent/child wiring
  • missing rendered links for source pages
  • duplicated scope between nearby pages
  • repeated confusions that deserve a synthesis page
  • rendered/source drift
  • weak browse entry points

Rule: a lint pass should improve retrieval, not just cleanliness.

Suggested operating cadence

  • During active work: ingest small findings as they happen and keep rendered pages current
  • After answering an important recurring question: ask whether the answer should become a durable page
  • Periodically: run a lint-style pass over one branch or cluster

When to create a synthesis page

Create a synthesis page when the answer requires combining multiple pages every time, a smaller model is likely to make the wrong jump, or the confusion is structural and recurring.

Recent good examples include kiosk-queue-shell-vs-content-owner and queue-interface-vs-queue-wizard.

Current status

This page makes the maintenance model explicit: the wiki is not only a documentation set. It is a maintained knowledge system with three core operations — ingest new truth, query for answers, and lint for health and retrieval quality.
