wiki structure notes

Purpose

This page defines how the now-playing knowledge wiki should be structured and expanded over time.

It is the maintenance/governance layer for the wiki.

Core structure model

A useful current model is:

  • hub page
  • parent page
  • child page
  • companion page
  • runbook page
  • reference/support page

These should not all be used interchangeably.

Hub pages

Hub pages orient the reader and route them into the right branch. They should explain the branch at a high level and link to the important child/related pages, but should not become exhaustive implementation catalogs.

Parent pages

Parent pages define a branch and provide the main conceptual frame for its children. They should define the branch clearly, explain boundaries, and keep the branch navigable.

Child pages

Child pages exist when a subtopic is important enough to deserve its own durable place under a parent branch.

A page deserves to become a child page when it answers a distinct practical question, has its own ownership boundary, is repeatedly referenced nearby, or would otherwise muddy the parent.

Companion pages

Companion pages are sibling pages that clarify the same branch from different angles. They should cross-link explicitly and avoid competing for the same exact scope.

Runbook pages

Runbooks are action-oriented troubleshooting or verification guides. If the main purpose is operational sequence, it should be a runbook, not a concept page.

Reference/support pages

Support pages centralize durable cross-cutting material without absorbing branch-specific ownership detail.

Parent/child versus see-also links

  • Use parent/child when the topic clearly belongs under a branch.
  • Use companion/sibling when pages explain the same branch from different angles.
  • Use see also when the relationship is contextual rather than structural.

Anti-pattern: do not use “see also” to hide a missing parent/child relationship.

Reachability rule

Every substantive page should be meaningfully reachable from at least one higher-level source page and at least one higher-level rendered page.

Source truth versus rendered truth

Markdown is the canonical truth, but the rendered wiki is the live browsing surface. If a source page is added or materially reframed, the rendered/live page should be updated accordingly.

Naming conventions

  • *-interface.md for surfaces or interface families
  • *-internals.md for implementation-aware subtopics
  • *-ownership.md for ownership/failure-path mapping
  • *-runbook.md for operational guides
  • *-overview.md for branch/system summaries
  • *-notes.md for maintenance/meta pages

Confidence language rule

The wiki should preserve the difference between verified truth, best current interpretation, and still-open questions. Do not flatten uncertainty into fake certainty.

Recommended maintenance checklist for new pages

  1. clear page type
  2. distinct practical question
  3. linked from a sensible higher-level source page
  4. rendered HTML updated
  5. rendered page reachable from a sensible higher-level rendered page
  6. parent updated if a child absorbed its detail
  7. confidence language matches proof level

Current status

This page gives the wiki an explicit structure/governance layer so the Phase 1 and Phase 2 work does not drift over time.
