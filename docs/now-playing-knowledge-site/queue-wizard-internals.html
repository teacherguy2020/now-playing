queue-wizard internals

Purpose

This page documents the current internal model of Queue Wizard in the now-playing system.

It turns Queue Wizard from an indirectly described concept into a more explicit internal model.

Important scope note

This page is intentionally honest about what is known versus what still needs route/file proofing.

What is already well supported by current wiki work:

  • Queue Wizard is a real queue-shaping layer above raw queue mutation
  • it has preview/apply semantics
  • it is used by substantive controller/now-playing surfaces
  • it intersects with Last.fm / Vibe features
  • it likely owns album-level and related higher-order queue actions

What is not yet fully locked down here:

  • the exact complete route list
  • the exact owning route file(s)
  • the exact helper/service chain behind every Queue Wizard action

Why Queue Wizard matters

Queue Wizard appears to be one of the strongest signs that now-playing is not merely a thin MPD remote. It adds queue shaping and curation above raw queue primitives.

High-level role

A good current interpretation is:

  • Queue Wizard is the queue-shaping and queue-curation layer of the system
  • it sits above raw transport and raw queue mutation
  • it is where feature-side intent becomes a more deliberate queue plan

Queue Wizard versus raw queue mutation

Raw queue mutation

Raw queue mutation means operations like append item(s), replace queue, and crop queue.

Queue Wizard

Queue Wizard adds a richer layer on top, including previewing queue changes before apply, album-level queue planning, and structured queue shaping rather than one raw append call.

Current known semantics

1. Preview / apply pattern

This is the strongest current signal. A good current interpretation is that Queue Wizard first computes a candidate queue change, supports preview, and then apply commits the change.

2. Album-level queue actions

Current wiki work already notes queue-wizard preview/apply calls for album actions. This suggests Queue Wizard is especially important for “build queue from this album” style operations.

3. Feature-driven queue shaping

Config/Last.fm work already notes that when Last.fm is enabled/configured, Vibe tools are available in Queue Wizard. That reinforces the idea that Queue Wizard is also part of discovery/curation-driven queue generation.

Current known caller surfaces

  • substantive controller-now-playing* pages
  • controller-side queue and now-playing interactions more broadly
  • Last.fm / Vibe discovery tooling

Queue Wizard in the overall playback model

A useful layered model is:

  • Layer 1: transport
  • Layer 2: raw queue mutation
  • Layer 3: queue inspection
  • Layer 4: queue shaping / curation — Queue Wizard

This makes Queue Wizard a Layer 4 concept.

Current route knowledge

Current wiki work supports the existence of:

  • a Queue Wizard route family
  • preview/apply semantics
  • album-level actions

But it does not yet fully prove the exact route names, request/response shapes, or owning file/module chain.

Relationship to YouTube queueing

YouTube queueing through POST /youtube/queue is a direct feature-specific queue submission path. Queue Wizard is better understood as a more general queue-shaping/curation layer.

Relationship to embedded queue surfaces

Embedded queue surfaces like controller-queue.html are mainly queue inspection/control UI. Queue Wizard is the higher-order queue-shaping logic that may feed into what the queue becomes.

Current status

This page gives the wiki a much sharper internal model of Queue Wizard: it is the queue-shaping layer, it uses preview/apply semantics, it is used by controller/now-playing surfaces, it intersects with discovery/curation features like Last.fm/Vibe, and it still needs exact route/file proofing later.
