deployment and ops

Purpose

This page explains what “deployment and operations” concretely means for the now-playing project.

A better current interpretation is:

  • deployment = getting changed code/config/runtime behavior onto the actual app host and, when relevant, the moOde host
  • ops = understanding which machine or service a change affects, how that effect becomes live, and what must be verified afterward

Concrete operational model

1. App-host code and static pages

This includes HTML pages, browser JS/CSS, route files, services, and app-host entry logic such as moode-nowplaying-api.mjs. These changes primarily affect the app host.

2. moOde-host display/runtime effects

This includes browser target changes, display mode selection behavior, Peppy/browser-target actions, and host-side display/runtime results triggered through runtime-admin routes.

3. Host override material

This includes moOde-side override scripts/configs and local host patches that are not automatically implied by app deploys.

What usually gets deployed

Frontend/static surface changes

Examples include app.html, controller.html, kiosk.html, visualizer.html, and CSS/browser-side JS changes. These usually mean updating app-host files and verifying the real affected surface; API restart is often unnecessary if the change is truly frontend-only.

Backend / route / service changes

Examples include src/routes/*.mjs, service helpers, and app-host state-truth endpoints. These usually mean updating app-host code, likely restarting or at least explicitly verifying runtime behavior, then checking affected surfaces.

Runtime-admin changes

Examples include /config/runtime/*, /config/moode/*, and /config/services/*. These can trigger immediate live actions and may affect the app host, the moOde host, or both.

Concrete decision rules

  • Frontend-only: deploy/update files, often no API restart, but verify in the real surface context.
  • Route/service logic: deploy/update app-host code, likely restart-sensitive, verify endpoint first, then UI/surfaces.
  • Config/runtime-admin logic: verify which machine is the real effect target and whether host-side visible behavior actually changed.
  • Host override material: normal app-deploy assumptions are not sufficient; verify real host state explicitly.

App host vs moOde host

App host

Primarily owns API routes, controller/display page serving, config/runtime handling, state-truth endpoints like /now-playing, and queue/playback mediation logic.

moOde host

Is where many display-target actions become visibly real: browser URL changes, display-mode switching, and host-side display/runtime behavior.

If the endpoint says success but the real screen still looks wrong, do not stop at the app host.

Restart reality

Not all changes become live the same way.

  • Often no restart needed: frontend-only page/style changes
  • Often restart-sensitive: route logic, service helpers, startup-captured config/env behavior, API-process changes

For concrete restart behavior, use restart-and-runtime-admin-troubleshooting and backend-change-verification-runbook.

Verification model after deployment

  1. Verify the correct layer: endpoint, page/surface, or host-side visible effect
  2. Verify the correct context: standalone, embedded, kiosk, moOde browser target, and mode-specific playback context when relevant
  3. Verify the real symptom: not just a nearby symptom
