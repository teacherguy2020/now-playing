It is meant as the bridge between project structure and real running behavior.

Documented operational areas

The project should be treated as an actively operated system, not just a code repository. Operational/admin topics include installation/setup, missing runtime dependencies, API-host-specific operation, configuration management, service control, runtime verification, current app URL checks, Chromium/process checks, patched endpoint checks, wake-display verification, and administration.

Deployment-related themes

  • environment preparation matters
  • missing host dependencies are a real concern
  • service management is part of normal operation
  • host-specific behavior exists
  • display/runtime state must sometimes be verified directly
  • local overrides can materially change what correct runtime behavior looks like

Repo-verified refinement: current operational control is more route/runtime-admin-centered than old helper-script-centered in several important areas, and src/routes/config.runtime-admin.routes.mjs is a major live operational surface.

Change classes and what they usually imply

Frontend / page / style changes

Usually implies no API restart, but still requires visible verification on the relevant surface and awareness of iframe/runtime/display-state edge cases.

Route / service / config / runtime-admin changes

Usually implies likely API restart or explicit runtime verification, plus endpoint verification and surface verification afterward.

Important restart-boundary refinement: some values are read per request or patched live into process.env, while others remain restart-sensitive because module-level config/env values were initialized at API boot.

Runtime-admin or host-control actions

Usually implies immediate host-side state changes are possible, so verify host/service/process state, not just API responses.

Host override changes

Usually implies a separate host-side application path and verification on the real moOde host after apply.

Runtime verification and service control

Operationally, future agents should expect to verify what is actually running, confirm runtime endpoints/URLs, check service/process state, verify display-specific behavior after changes, distinguish frontend-only changes from backend/integration changes, and distinguish app-runtime changes from SSH-backed moOde/host-control actions.

Local-environment dependence

This project has meaningful local-environment dependence. Operational behavior depends on which host is primary, which host owns display/runtime behavior, local patches/overrides, the preferred deploy target, and whether behavior is coming from stock code or environment-specific modifications.

Operational cautions

  • do not assume frontend changes are the whole story
  • do not assume documented behavior is identical to local behavior when local patches exist
  • verify before and after changes when touching runtime-sensitive areas
  • check local-environment guardrails before deploys or host-targeted changes
  • treat display, wake, endpoint, and service issues as potentially cross-cutting problems
