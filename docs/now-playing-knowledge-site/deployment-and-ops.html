deployment and ops

Purpose

This page explains how changes in now-playing actually become live.

It should answer: what did I change, which machine or process does that affect, does it usually need only refresh/verification or also restart, and what should I verify afterward?

The main operational lesson

A change can succeed at one layer and still fail to produce the effect you actually care about.

Examples: an endpoint can return success while the moOde display target did not change; a page can look right in a normal browser but still be wrong in kiosk or embedded mode; config can be saved while startup-captured behavior has not changed yet.

The system has more than one operational plane

1. App host plane

Owns API routes, served UI, controller/display pages, state-truth endpoints like /now-playing, and queue/playback mediation logic.

2. moOde host effect plane

Owns visible display-target results, browser-target switching, and some runtime/display effects triggered through control-plane routes.

3. Host override plane

Owns manual or mirrored host-side override material outside the normal app tree.

Change type → effect → verify

If you changed a browser page or frontend-only asset

Examples: controller.html, app.html, kiosk.html, visualizer.html, CSS, browser-side JS.

Usually means: app-host served files changed; refresh is often enough; API restart is often unnecessary if the change is truly frontend-only.

But you still must verify the real operating context: normal, embedded, kiosk, or moOde-pushed target.

If you changed route logic or service logic

Examples: src/routes/*.mjs, service helpers, moode-nowplaying-api.mjs.

Usually means: app-host behavior changed, route verification should come first, and restart may be needed depending on how the process loads the code.

If you changed runtime-admin behavior

Examples: /config/runtime/*, /config/moode/*, /config/services/*.

Usually means: you changed control-plane behavior that may affect the app host, the moOde host, or both. Verify the actual target effect, not only the response.

If you changed a state-truth endpoint

Examples: /now-playing, /next-up, Alexa state helpers.

Usually means: you changed one of the system’s central truth surfaces. Verify the endpoint first, then verify the dependent surfaces.

If you changed a host override or host-side patch

Usually means: normal app deploy assumptions are not enough. Verify the real host state and the real effect on that host.

App host vs moOde host: the key question

When something changed and the visible system still looks wrong, ask: Which machine should have changed?

  • App host: route changed, served page changed, /now-playing output changed, controller/kiosk logic changed
  • moOde host: browser target changed, room-facing display switched, host-side display/runtime action took effect

Restart reality

Not every change needs restart, but route/service/startup-captured behavior is more restart-sensitive than page-only changes.

For more detailed restart reasoning, use restart-and-runtime-admin-troubleshooting and backend-change-verification-runbook.

What to verify after common change types

Example: changed controller.html

Likely effect: app-host-served UI changed. Verify the real controller/kiosk context, not only a generic browser load.

Example: changed visualizer.html

Likely effect: visualizer engine behavior changed. Verify fullscreen, embedded, and push-to-moOde behavior if relevant.

Example: changed moode-nowplaying-api.mjs

Likely effect: state-truth endpoint behavior changed. Verify the endpoint directly, then verify downstream controller/display surfaces.

Example: changed config.runtime-admin.routes.mjs

Likely effect: runtime-admin/control-plane behavior changed. Verify both endpoint response and real target effect.

Example: changed moOde-side override material

Likely effect: host behavior changed outside the normal app tree. Verify live host state, not just mirrored repo content.

What this page is really teaching

  • there is more than one operational plane in the project
  • app host and moOde host should not be mentally collapsed
  • different change types become live in different ways
  • the first verification question is often “what layer should have changed?”
  • successful response, served file, and visible effect are three different things
