api service overview

Purpose

This page introduces the API service layer in the now-playing system.

It exists because the wiki already documents many API calls indirectly through interface and feature pages, but it also needs a dedicated API-centered parent page.

Why this page matters

A lot of the system’s real behavior is not owned only by HTML pages or frontend scripts. It is mediated through an app-host API service that serves config and runtime state, accepts privileged maintenance actions, exposes playback and queue controls, powers feature-specific bridges like YouTube and radio evaluation, and coordinates with MPD, moOde, and other runtime-adjacent services.

High-level role of the API service

A good current interpretation is:

  • the API service is the app-side control plane of the system
  • frontend/controller/admin pages call into it over HTTP
  • it is one of the main places where UI intent turns into operational behavior

That makes it more than backend plumbing. It is one of the project’s primary architectural surfaces.

What the API service appears to own

  • config and runtime state
  • environment verification and maintenance
  • playback and queue actions
  • ratings database maintenance
  • YouTube search/resolve/playlist/queue flows
  • radio metadata debug/evaluation flows
  • Alexa domain verification and related setup checks
  • service-control actions such as mpdscribble control
  • moOde browser-target inspection and related display-control helpers
  • podcast automation/status helpers

Relationship to the rest of the system

The API service sits between UI/surface pages, integration boundaries like MPD/moOde/YouTube/Alexa, and the operational/runtime layer where track-key-protected maintenance calls, SSH-backed verification, and restart-sensitive behavior live.

Request model

  • HTTP GET and POST are both used heavily
  • many privileged or maintenance-sensitive operations require a track key
  • the track key commonly appears as x-track-key
  • header-based auth matters enough to be called out explicitly in the integration docs

Current route-family map

Config / runtime family

Examples already documented elsewhere include route families under /config/runtime, /config/alexa/*, /config/ratings/*, /config/services/*, and /config/moode/*.

Playback / queue / control family

Current wiki pages already show playback/control calls used by controller and now-playing surfaces, and that family now has a stronger companion page in api-playback-and-queue-endpoints.md.

YouTube integration family

Current wiki coverage already shows /youtube/search, /youtube/playlist, /youtube/resolve, and /youtube/queue.

Radio debug / evaluation family

Current wiki coverage already shows /debug/radio-metadata-log and /debug/radio-metadata-log/clear.

Podcast automation family

Current config work already shows /podcasts/nightly-status and /podcasts/nightly-run.

Architectural interpretation

A good current interpretation is that the API service is not just a generic REST surface. Route families often map directly to feature ownership and operational workflows, and many routes appear to be first-class ownership points for system behavior.
