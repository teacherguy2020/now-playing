controller kiosk mode

Purpose

This page documents what controller.html actually does when kiosk mode is active.

Direct file inspection shows that controller.html is a real kiosk implementation surface, owning kiosk layout activation, kiosk editor mode, right-pane hosting, child-page remapping, and controller-profile/theme application during kiosk use.

Kiosk mode is explicit in controller.html

Direct source evidence shows the page defines:

  • IS_PREVIEW = ...preview=1
  • IS_KIOSK_1280 = ...kiosk=1 || profile=1280x400 || layout=1280x400
  • IS_KIOSK_EDITOR = IS_KIOSK_1280 && (...edit=1 || ...designer=1)

When kiosk mode is active, it adds body.kiosk1280 and optionally body.kioskEditor.

Kiosk layout is embedded directly in the page

Direct file inspection shows extensive CSS for kiosk-specific controller behavior, including:

  • body.kiosk1280
  • body.kiosk1280:not(.kioskEditor)
  • #kioskCanvas
  • .recentWrap
  • .kPane

This supports a 1280×400 live kiosk profile, a distinct editor mode, and a slide-in right-pane layout.

Concrete kiosk DOM anchors

Direct file inspection shows:

  • #kioskCtl
  • #kioskSizeSel
  • #kioskPushBtn
  • #kioskPushStat
  • #kioskCanvas
  • #kioskPane
  • #kioskPaneFrame

So the page contains both kiosk editor controls and live pane infrastructure directly.

Kiosk editor mode is real and separate

The page contains initKioskControls(), which immediately exits unless IS_KIOSK_EDITOR is true.

Observed behavior includes using #kioskSizeSel, wiring #kioskPushBtn, and updating #kioskPushStat.

This means controller-side kiosk mode contains both live presentation behavior and a distinct editor/designer branch.

Controller profile integration during kiosk use

Direct file inspection shows:

  • MOBILE_PROFILE_KEY = 'nowplaying.mobile.profile.v2'
  • MOBILE_DEFAULT_PROFILE including devicePreset, theme, layout, recentSource, and colorPreset

The page explicitly applies URL-driven overrides for:

  • devicePreset
  • theme
  • colorPreset
  • recentSource
  • custom color fields

It also defines loadServerProfile() using /config/controller-profile, but skips that path when IS_PREVIEW or local-profile-only mode is active. That matters because the kiosk handoff sets preview=1.

Kiosk right-pane behavior lives here too

Direct file inspection shows:

  • #kioskPane
  • #kioskPaneFrame
  • closeKioskPane()
  • message handling for np-kiosk-hide-pane

The controller-side routing logic:

  • detects kiosk/visualizer child targets
  • maps kiosk-named child pages into controller-family pages
  • loads them into #kioskPaneFrame
  • closes the pane when the same row is tapped again in pane mode
  • navigates directly when not using right-pane mode

Observed mappings include kiosk-now-playing.html, kiosk-playlists.html, kiosk-artists.html, kiosk-albums.html, kiosk-genres.html, kiosk-podcasts.html, kiosk-radio.html, kiosk-queue-wizard.html, and kiosk-queue.html into controller-family views.

What controller.html is not

  • It is not the initial entry bridge — kiosk.html owns that.
  • It is not the separate scaffold/prototype path — controller-kiosk.html owns that.
  • It is not every child content surface itself — embedded child pages still own their own content.

Best current interpretation

controller.html is one of the main real kiosk implementation pages in the repo. It explicitly detects kiosk and kiosk-editor modes, owns the kiosk layout shell and right-pane behavior, applies controller-profile/query-driven state, and serves as the practical destination for the live kiosk handoff from kiosk.html.

Current status

This page is now much less abstract: the code strongly supports treating controller.html as a real kiosk-mode implementation surface, not merely a generic destination after redirect.
