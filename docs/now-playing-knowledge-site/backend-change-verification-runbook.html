backend change verification runbook

Purpose

This runbook explains what to verify after backend-facing now-playing changes: route changes, service/helper changes, config/runtime behavior changes, runtime-admin behavior changes, host-control side effects, and backend-supported UI behavior.

Core rule

After a backend-facing change, verify in this order:

  1. what class of change was made
  2. whether restart sensitivity is likely
  3. which host should show the real effect
  4. which endpoints should be checked directly
  5. which user-facing surfaces should still be verified

Step 1: classify the change

  • Frontend/page/style only — often no API restart, but visible verification still required.
  • Route/service/config-consuming backend logic — endpoint verification required, restart may be needed.
  • Runtime-admin or host-control behavior — endpoint success is not enough; verify real host-side effect.
  • Host override changes — verify on the real moOde host after apply.

Step 2: decide whether restart is likely

Restart is more likely when backend code changed, startup-captured config/env behavior changed, or the route warns that env-overridden values need restart to apply.

Important clue: /config/runtime writes config to disk but env-overridden values may still require restart with --update-env.

Step 3: identify where the real effect should happen

  • App host (10.0.0.4) — API behavior, route/service logic, config writes, restart-sensitive backend behavior.
  • moOde host (10.0.0.254) — browser URL changes, display mode changes, PeppyMeter/PeppyALSA actions, host-side runtime-admin effects.

Step 4: verify the endpoint or direct backend contract

Check the changed endpoint directly, then any nearby diagnostics/status endpoint that exposes the same path or effect.

Important guardrail: a 200 response proves the route handled the request, not necessarily that the real-world state now matches the intended result.

Step 5: verify the user-facing surface anyway

After backend verification, check the relevant surface: app.html, controller pages, display.html, index.html, peppy.html, player-render.html/player.html, or kiosk/controller surfaces as applicable.

Step 6: verify historically fragile areas when relevant

  • Display/runtime changes: also check iframe vs top-level differences and router vs final render target behavior.
  • Playback/art/status changes: also check mode classification and last-good/fallback behavior.
  • Next Up / queue / Alexa changes: also check queue-head truth, /next-up, and controller cards/views.
  • Radio metadata changes: also check holdback, suppression, fallback logo/art, and whether the new result is actually better.

Step 7: choose the minimum required verification set

  • Frontend-only: surface refresh + visible verification.
  • Backend route/service: endpoint verification + restart if needed + UI verification.
  • Runtime-admin/host-control: route response + host-side state + visible surface verification.
  • Host override: applied-on-host verification + live behavior verification.

Step 8: declare success only when the right layer agrees

A change is only really verified when the relevant layers agree: endpoint, host-side effect when applicable, and visible user-facing behavior.
