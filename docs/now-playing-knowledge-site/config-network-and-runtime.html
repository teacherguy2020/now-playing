config network and runtime

Purpose

This page documents the Network & Runtime portion of now-playing/config.html.

This card is the foundational bootstrap layer for the rest of the Config page. It establishes where the API and UI live, how MPD and moOde are reached, what track key protects privileged actions, what library/mount paths the system assumes, and whether the runtime environment is valid enough to unlock other modules.

Main visible controls

  • apiNodeIp
  • apiPort
  • uiPort
  • mpdHost
  • mpdPort
  • moodeSshHost
  • moodeSshUser
  • moodeBaseUrl
  • configTrackKey
  • pathMusicLibraryRoot
  • pathMoodeUsbMount
  • pathPiMountBase
  • checkRuntimeBtn
  • runtimeCheckStatus
  • runtimeCheckDetails
  • sshSetupCmds

Defaults and prefill behavior

applyInitialNetworkPrefills() shows that the page does conservative environment-aware defaulting:

  • API port defaults to 3101
  • UI port defaults to 8101
  • MPD port defaults to 6600
  • music library root defaults to /var/lib/mpd/music
  • API host prefers nowplaying.local when reachable, otherwise falls back to the current host

MPD / moOde SSH setup

This block separates:

  • MPD host/port
  • moOde SSH host/user
  • optional moOde base URL override

That reflects the fact that app host, MPD host, moOde SSH endpoint, and moOde web endpoint are related but not always identical.

Track key

configTrackKey is cached locally and used to protect privileged maintenance/config endpoints through x-track-key.

Paths and mount assumptions

The card manages:

  • music library root
  • moOde USB mount path
  • API-host mount base
  • podcast root

derivePathsFromMoode() can infer the API-host mount and podcast path from the moOde-side media path, which shows the page understands a mapping between the two environments.

SSH setup help

refreshSshSetupHelp() generates one-time helper commands such as:

  • ssh-keygen
  • ssh-copy-id
  • basic SSH verification
  • sudo non-interactive verification

This is a clear sign that the page is trying to help the operator reach an automation-ready state, not just save values.

Runtime verification

checkRuntimeEnv() is the key active workflow in this card.

It sends:

  • POST /config/runtime/check-env

with MPD host/port, SSH host/user, and path values. It then updates status text, details, lights, pills, and runtime gating state based on returned SSH/MPD/path results.

Gating / unlock behavior

The page treats this card as the “Step 1” bootstrap layer and uses runtime readiness to gate other modules. That is one of the strongest signals that Network & Runtime is the operational control-plane card for Config.

Podcast-root creation

When Podcasts is enabled and the podcast root is missing, the runtime details area can offer a fix action implemented by:

  • POST /config/runtime/ensure-podcast-root

This is a good example of the runtime card actively fixing environment issues rather than only reporting them.

Current status

This card is best understood as:

  • environment bootstrap
  • security setup
  • path model definition
  • SSH help surface
  • runtime verification console
  • gating/unlocking layer for the broader Config page
