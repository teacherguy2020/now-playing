queue and playback model

Purpose

This page establishes the current conceptual model for playback and queue behavior in the now-playing system.

It exists because playback and queue behavior are some of the most important remaining high-value knowledge gaps in the wiki.

Why this page matters

A lot of practical confusion in this project comes from mixing together several things that are related but not identical:

  • transport control
  • queue inspection
  • queue mutation
  • queue shaping/curation
  • playback-mode-specific authority
  • display/presentation side effects

This page is the foundation for separating those concepts clearly.

High-level model

A good current interpretation is:

  • playback control is about the current transport state and what is actively playing now or next
  • queue control is about what items are present and in what order
  • queue shaping is a richer layer above raw queue mutation, where the system previews, filters, or restructures what should be played
  • controller surfaces usually initiate these actions
  • app-host API routes are usually the control plane through which those actions become real
  • MPD remains a core playback/queue truth source, but the app frequently mediates, enriches, or translates what that truth means for visible behavior

Core distinction: playback vs queue

Playback control

Playback control means actions such as play, pause, resume, next/skip, previous/back, and other direct transport-style actions.

Current wiki work already shows direct playback route usage such as:

  • /config/diagnostics/playback

A good current interpretation is that playback control acts on current transport/session state rather than primarily restructuring the whole queue.

Queue control

Queue control means operations such as:

  • inspect queue contents
  • append an item or items
  • replace the queue
  • crop/truncate a queue variant
  • load a set of items derived from a search, album, playlist, or other selection

Current wiki work already shows:

  • YouTube queue submission via /youtube/queue
  • queue modal / embedded queue loading through controller-queue.html
  • queue-wizard preview/apply flows for album-level actions

Queue shaping / curation

This is the richest layer above raw queue mutation. It includes previewing queue plans, replacing/cropping intentionally, and building a queue from higher-level feature flows rather than one raw append action.

Current actor model

  1. User-facing surfaces initiate intent — controller-now-playing pages, mobile/tablet/controller shells, YouTube page, queue surfaces, kiosk-embedded queue/now-playing actions.
  2. App-host routes mediate the action — accepting the action, translating it into queue/playback semantics, coordinating with MPD and related helpers, and sometimes mixing in additional app-side logic.
  3. MPD remains an important underlying truth source — but visible playback and queue behavior is often app-mediated rather than raw-MPD-only.
  4. Visible surfaces re-render from resulting state — often through route-backed derived state rather than a direct raw MPD dump alone.

Current known playback/queue route families

Direct playback family

  • /config/diagnostics/playback

Queue-wizard family

Current evidence repeatedly references preview/apply flows, though the exact route list is still not fully enumerated.

YouTube queue family

  • POST /youtube/queue

with queue modes including append, crop, and replace.

Embedded queue surface

  • controller-queue.html
  • controller-queue.html?embedded=1

This tells us queue inspection/control is a first-class surface even though its backend route family still needs deeper proofing.

Display/presentation overlap family

  • POST /config/moode/browser-url
  • POST /peppy/last-profile

These are not pure playback routes, but they sit close enough to operational playback/presentation control that they belong adjacent to the model.

Current known queue mutation patterns

  • Direct append — add content to the existing queue
  • Replace — replace current queue contents with a new set
  • Crop — a queue-shaping variant explicitly surfaced in YouTube queue submission
  • Preview/apply — richer queue-wizard pattern where the operator/user previews a queue shape before committing it
  • Embedded queue inspection — view queue state from controller/now-playing contexts via embedded queue surfaces

Current status

This page gives the wiki the foundational distinction it was missing:

  • playback control
  • queue control
  • queue shaping/curation
  • controller/API/MPD roles
  • display/presentation overlap

That is enough to make the rest of the Phase 2 work more coherent.
