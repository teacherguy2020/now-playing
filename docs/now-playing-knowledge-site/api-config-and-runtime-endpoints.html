api config and runtime endpoints

Purpose

This page documents the config- and runtime-oriented endpoint families exposed by the app-host API service.

It consolidates routes that were previously documented only indirectly through Config and feature pages.

Auth / protection model

A major repeated pattern in this endpoint family is:

  • privileged operations require a track key
  • the track key is sent as x-track-key

So many of these routes are maintenance/control endpoints, not just convenience endpoints.

Route family overview

1. Runtime config load/save

Centered on:

  • /config/runtime

This route family is used to load current runtime config, save structured config changes, and save full raw config via the advanced JSON editor.

2. Runtime environment verification

Centered on:

  • POST /config/runtime/check-env

Used to validate MPD connectivity, SSH connectivity to moOde, and path existence/health.

3. Runtime repair: podcast-root creation

Includes:

  • POST /config/runtime/ensure-podcast-root

This is an environment-repair route, not just inspection.

4. Alexa provisioning check

Includes:

  • POST /config/alexa/check-domain

Used to verify whether the configured Alexa public domain is reachable.

5. Ratings maintenance family

Includes:

  • GET /config/ratings/sticker-status
  • GET /config/ratings/sticker-backups
  • POST /config/ratings/sticker-backup
  • POST /config/ratings/sticker-restore

These routes support ratings DB verification, backup listing, backup creation, and restore.

6. Service-control family

Current documented example:

  • GET /config/services/mpdscribble/status
  • POST /config/services/mpdscribble/action

This family shows that the config/runtime API also controls live services, not just stored settings.

7. moOde browser-target / display-control family

Current documented example:

  • GET /config/moode/browser-url/status

Used to inspect whether the moOde browser target points at the expected display URL.

8. Podcast automation family that overlaps config/runtime use

Includes:

  • GET /podcasts/nightly-status
  • POST /podcasts/nightly-run

These are not under /config/, but they belong near this family because current Config automation flows depend on them.

Current status

This page gives the wiki an API-centered map of the config/runtime control plane instead of leaving those endpoints scattered across feature pages.
