display interface

Purpose

This page documents the display branch of the now-playing ecosystem.

A better current interpretation than the older version is:

  • the display branch is not just “things that look visual”
  • it is the family of surfaces that turn app-host state into room-facing/browser-facing presentation
  • much of that presentation branches outward from /now-playing as a central visible-truth hinge
  • the branch already has real file-backed subpages for launch surfaces, kiosk shell behavior, wrappers, embedded display behavior, and controller-adjacent visual surfaces

So this page should now act as a display branch hub, not a placeholder taxonomy page.

Why this page matters

Display behavior is one of the easiest places for agents to make wrong assumptions. A display problem might actually belong to state truth, kiosk shell routing, embedded child-page behavior, moOde browser-target selection, display-mode selection via peppy/last-profile, or runtime/host-specific display behavior.

Central display truth starts from /now-playing

One of the most important current architectural facts is that the independent display/controller system branches outward from the app-host’s /now-playing state surface.

That endpoint is one of the project’s central visible-truth hinges. It is where mode-aware playback state becomes the payload that display/controller surfaces consume.

The display branch should therefore be read together with api-state-truth-endpoints.md and playback-authority-by-mode.md.

What belongs in the display branch

A useful terminology split is:

  • display mode / renderer = Player, Peppy, Visualizer
  • presentation/shell mode = Kiosk
  • launch/helper surface = pages such as displays.html, controller-visualizer.html, index1080.html
  • designer surface = pages such as player.html, peppy.html, kiosk-designer.html
  • embedded visual surface = controller-hosted child pages and iframe-driven visual behavior
  • skin = a visual/theme variant within a broader display mode

Strong current branch map

1. Central visible-state truth surfaces

Start with api-state-truth-endpoints and playback-authority-by-mode.

2. Display-mode / renderer flows

Player, Peppy, and Visualizer are distinct from kiosk shell mode. Their live selection/push behavior is materially tied to display-launch-and-wrapper-surfaces, where displays.html records the chosen mode via /peppy/last-profile and pushes a shared display.html?kiosk=1 target.

3. Kiosk as presentation/shell mode

Kiosk is explicitly a shell/presentation mode, not just another renderer. Strong current anchor pages are kiosk-interface, kiosk-launch-and-routing, and controller-kiosk-mode.

4. Launch / wrapper / adapter surfaces

This is now a real documented subfamily, not a vague future branch. display-launch-and-wrapper-surfaces now proves that displays.html is a moOde browser-target push console, controller-visualizer.html is a thin host for visualizer.html, and index1080.html is just a redirect shim.

5. Embedded visual behavior inside controller flows

Display behavior also appears in controller-hosted contexts. Strong anchor pages include visualizer-in-embedded-mode, embedded-pane-contracts, now-playing-surface-variants, and controller-now-playing-anatomy.

6. Runtime / host / display-target behavior

Display behavior is inseparable from host/runtime reality. Strong companions here are api-config-and-runtime-endpoints, deployment-and-ops, and local-environment.

What this branch is now confident about

  • /now-playing is the central state hinge for much of the visible display ecosystem
  • display modes (Player/Peppy/Visualizer) are distinct from kiosk shell mode
  • displays.html is a real launch/push console for live display target selection
  • controller-visualizer.html is a wrapper host, not a renderer implementation
  • kiosk.html is not the live kiosk shell; it is a profile-sync + redirect bridge
  • controller.html owns real kiosk shell behavior
  • many kiosk child routes are aliases into controller-family pages
  • display behavior is often downstream of playback truth, embedded-pane rules, and runtime host behavior

High-value starting paths

  • If visible display content is wrong: start with api-state-truth-endpoints, then playback-authority-by-mode, then the relevant display/kiosk child page.
  • If the wrong display target is being shown on moOde: start with display-launch-and-wrapper-surfaces, then api-config-and-runtime-endpoints, then local-environment.
  • If kiosk presentation behavior is wrong: start with kiosk-launch-and-routing, then controller-kiosk-mode, then kiosk-right-pane-routing.
  • If embedded visual behavior is wrong: start with embedded-pane-contracts, then visualizer-in-embedded-mode, then controller-now-playing-anatomy.

Current status

It is no longer just a placeholder map for future drill-downs. The current wiki already supports a stronger claim: the display branch has real file-backed subpages, the central visible-state hinge is /now-playing, and the main branch split is now explicit between state truth, display modes, kiosk shell mode, launch/wrapper surfaces, embedded visual behavior, and runtime/display-host behavior.
